## 🎯 這頁在教什麼

AI 會犯錯。

不是「偶爾當機」那種犯錯，是更麻煩的一種：它會**很有自信地講錯話**，甚至編造出根本不存在的東西——例如很篤定地說用了某個函式、某個檔案，但你的專案裡根本沒有這個東西。

這種現象有個名字，叫「幻覺」（hallucination）。

這頁不是要你因此不敢用 Claude Code，而是教你三個具體的動作，讓你用得安心：知道什麼時候該查、怎麼查、查什麼。

## 🎭 為什麼它會講得很篤定，卻是錯的

關鍵在於：AI 本質上是在**預測「接下來最合理的一句話」**，不是在**查證事實**。

打個比方：它比較像一位讀過非常多書、反應超快的人，被問問題時憑印象脫口而出——不是像查字典或資料庫，一筆一筆核對過再回答。

憑印象講出來的話，通常聽起來很順、很有條理，這是因為它很會「講得像一回事」，不代表它真的核對過。所以：

- 語氣篤定 ≠ 內容正確。
- 它不會主動說「這段我不太確定」——除非你要求它這樣做。
- 它說「做好了」，代表它覺得看起來像做完了，不代表它真的驗證過結果。

:::caution[常見情境：編出一個不存在的函式]
舉個典型的例子：你請 Claude 幫你檢查某段程式碼，它回答「這裡呼叫了 `formatCurrency()` 這個工具函式，應該沒問題」。

聽起來很有道理，對吧？問題是：你的專案裡根本沒有 `formatCurrency()` 這個函式，是 Claude 覺得「這種情境通常會有這樣一個函式」，就順著講出來了。

這不是它「壞掉」，是它在做它本來就在做的事——預測合理的下一句話。抓出這種狀況，靠的不是它自己講出來，是**你去對照原始程式碼**。
:::

## ✅ 三個一定要做的驗證動作

### 要可驗證的證據

「做好了」是一句話，不是證據。

有實際依據的說法長這樣：測試跑過、結果貼出來給你看；官方文件的連結附上，你可以自己點開；螢幕截圖跟原始設計稿並排比對。

技巧是：**在請它做事的時候，就先講清楚你要看到什麼證據**，而不是等它說完再回頭懷疑。細節可以看
[怎麼跟 Claude 說話](../../learn/how-to-talk-to-claude/)裡「怎麼驗收」的部分。

### 自己抽查對照

Claude 給的摘要、結論，是它消化過的版本，不是原始資料本身。

消化的過程就有可能夾帶錯誤或過度簡化。所以重要的東西，值得你自己回頭點開原始檔案看一眼、自己點開它附的連結確認內容真的有講到那件事。

抽查不用每次都全部查，抓重點、抓你最在意的那幾項就好。

### 越重要的決策越要查

不是每件事都要用同一個力道查證。

整理一份筆記格式錯了，改掉就好，成本很低。但要花錢、要對外發布、會影響到別人的決定——這種事出錯的代價高，值得你多花五分鐘人工複核，而不是看到「做好了」就直接往下一步走。

## 👉 跟著做

實際體驗一次「查證」的感覺：

<Steps>

1. 開一個新對話，貼上這句提示：

   ```text
   Claude Code 的 /rewind 指令是做什麼用的？
   請附上你查到的官方文件連結，不要只憑印象回答。
   ```

2. 收到回答後，自己點開它附的連結，確認內容真的有提到 `/rewind`，說法跟它講的一致。

3. 如果它答不出連結、或連結內容根本沒講到 `/rewind`，回問它：「這個說法有實際來源嗎？」看它怎麼回應。

</Steps>

**預期結果**：Claude 附上一個你可以自己點開查證的連結，而不是只用文字保證「這是對的」。

<details>
<summary>做壞了怎麼辦</summary>

如果它附的連結打不開、或內容根本沒講到你問的東西——不用緊張，這其實是一次「幻覺被你抓到」的成功案例，代表這個驗證習慣有用。

直接跟它說：「這個連結沒有講到 `/rewind`，你確定這個說法對嗎？」讓它重新查一次，而不是自己腦補一個答案接受下去。
</details>

## 📝 該進你的 CLAUDE.md 嗎？

如果你發現自己每次都要提醒 Claude「附證據」「附來源連結」，這種一直重複的要求，就值得寫進 CLAUDE.md，例如「做完任何檢查或修改，附上實際跑出來的結果，不要只回報結論」。

寫進去之後，每次對話自動生效，不用每次手動提醒。怎麼開始寫，看
[打造你自己的 CLAUDE.md](../../learn/your-own-claude-md/)。
