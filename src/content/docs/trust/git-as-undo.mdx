## 🧭 先備知識

不用會打任何 git 指令，這頁教你的是「用講的」讓 Claude 幫你存檔、回檔。

還沒摸過 Claude Code 的話，先看[新手上路](../../learn/getting-started/)；想知道哪些東西不該讓 AI 看到，看[不該分享的東西](../../trust/what-not-to-share/)。

## 🎮 git commit 就像遊戲的存檔點

小時候玩遊戲，過關前會先找個地方存檔——等一下不小心死掉，不用從頭砍起，讀檔回到剛才那一點就好。

git 的 commit 做的是同一件事，只是存的不是遊戲進度，是你資料夾裡所有檔案在某一刻的樣子。

commit 一次，就是「存一次檔」；反悔，就是「讀檔」回到某一次 commit 當下的狀態。

記住這句就夠了：**動手做比較大的改動之前，先請 Claude 幫你存個檔**。存檔不用錢、不花時間，卻能讓你之後放心讓它大膽嘗試。

## 🛟 較大改動前，先存個檔

<Steps>

1. 開始一個比較大的改動之前（例如「幫我重新排版整個側欄」「幫我把這十個檔案都改名」），先跟 Claude 說：

   ```text
   在動手之前，先幫我用 git 存一個檔（commit），
   訊息隨便寫清楚一點就好。
   ```

2. Claude 會跑幾個 git 指令把目前狀態存下來（如果這個資料夾還沒設定 git，它也會先幫你設定好）。跟其他會改動狀態的操作一樣，Claude Code 預設會先跳出視窗問你「可以嗎？」，確認一次就好。

3. 存檔完成，再讓它動手做原本要做的大改動。

</Steps>

**你會看到**：Claude 回報已經 commit，通常還會附一句它幫你寫的存檔說明。

## 🗣️ 不用背指令，直接用講的

git 本身的指令不好記（`add`、`commit`、`push`、revert……），但你完全不需要背。跟 Claude 講人話就好，它會自己轉成對的指令：

| 你想做的事 | 你可以直接說 |
|---|---|
| 存檔 | 「幫我用 git 存一個檔」 |
| 看之前存過哪些檔 | 「幫我看一下之前存過哪些版本」 |
| 反悔，回到剛才那個版本 | 「幫我回到剛才那個版本，這次的改動我不要了」 |
| 只想看差異，還沒決定要不要反悔 | 「幫我看一下這次改了什麼，跟上一次存檔比」 |

<Card title="它會先問過你" icon="approve-check">
commit、回到舊版本這類會改動狀態的操作，都算 Claude Code 眼中「有風險」的動作，動手前一樣會先跳出視窗問你，不會偷偷做。
</Card>

## 🆚 跟 Claude 自己的「回上一步」不一樣

Claude Code 自己也有一個內建的回溯功能（按兩下 `Esc`，或打 `/rewind`），聽起來很像，但跟 git 存檔不是同一件事：

<CardGrid>
	<Card title="/rewind：暫時的" icon="information">
		只在這次對話存在期間有用，而且只認得住 Claude 親手改過的檔案——它自己跑指令造成的改動（例如刪檔、搬檔），不算在內，救不回來。
	</Card>
	<Card title="git commit：長久的" icon="approve-check">
		不管過多久、換幾次對話，只要存過檔，都回得去。官方文件也是這樣定調的：把 `/rewind` 想成「暫時的救急」，git 才是「正式的存檔」，兩個搭配用，不是二選一。
	</Card>
</CardGrid>

## ⚠️ 老實講：它救不了什麼

:::caution[git 不是萬能保險]
- **沒存過檔的東西，救不回來。** 如果在請 Claude commit 之前就做了大改動，出事了也沒有版本可以讀回去——所以「先存檔、再動手」的順序不能倒過來。
- **已經傳到公開 repo（例如公開的 GitHub）的東西，等於已經公開了。** 就算之後刪掉那次 commit、甚至砍掉整個檔案，都不保證沒人已經看過、存過一份備份——一旦公開流出，很難真的收回來。什麼東西一開始就不該進 commit（密碼、私人資料等），看[不該分享的東西](../../trust/what-not-to-share/)。
:::

## ✍️ 跟著做

<Steps>

1. 找一個你在練習用的資料夾（不是重要的正式檔案），開 Claude Code。

2. 跟它說：

   ```text
   幫我用 git 存一個檔，訊息寫「練習存檔點」。
   ```

   看它怎麼回覆、跳出的確認視窗長什麼樣子，選允許。

3. 隨便改一個檔案內容（或請 Claude 幫你改），確認改動明顯看得出來。

4. 跟它說：

   ```text
   我不要剛才那個改動了，幫我回到上一個存檔點。
   ```

</Steps>

**預期結果**：檔案內容回到你第 2 步存檔當下的樣子，剛才那個改動不見了。

<details>
<summary>做壞了怎麼辦</summary>

這整頁教的就是「做壞了怎麼辦」的解法本身——真的卡住，最直接的辦法就是跟 Claude 說：「這次的改動有問題，幫我回到上一個存檔點」，讓它自己處理 git 的細節。

如果它回答「找不到可以回去的存檔點」，代表這次改動之前沒存過檔，這正是為什麼「先存檔、再動手」的順序很重要。
</details>

## 📝 該進你的 CLAUDE.md 嗎？

如果你發現自己常常忘記先存檔就讓 Claude 做大改動，可以把這個習慣寫進 CLAUDE.md，例如「做涉及多個檔案的改動之前，先主動幫我 git commit 一次」。寫進去之後，Claude 每次都會記得提醒你，不用你自己每次想起來才講。
