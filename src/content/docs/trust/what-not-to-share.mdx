## 🎯 這頁要幹嘛

一句話：有些東西一旦講出去、貼出去，就收不回來了。

這頁不是要嚇你不敢用 AI。是要讓你養成一個習慣：貼任何東西給 Claude 之前，先花三秒鐘想一下「這如果外流會怎樣」。

想得出答案，就繼續。想不出來、或答案是「慘」，先停一下，看看下面該怎麼做。

## 😬 四個真的會發生的地雷情境

### 情境一：把整包 `.env` 貼進對話框問「這是哪裡設定錯了」

你在 debug，圖方便把 `.env` 整個檔案內容複製貼上，想讓 Claude 幫你看哪一行有問題。裡面除了那個出錯的設定，還夾著資料庫密碼、第三方服務的 API key。

**風險**：這些內容整段進了對話紀錄。就算你自己電腦上沒事，只要這段對話被截圖、轉發、或哪天不小心分享出去，裡面的金鑰就跟著曝光了——而且你原本只是想問一行設定，卻不必要地把整包密碼都送了出去。

**該怎麼做**：只貼「出問題的那一行」，把值本身換成 `XXXXX` 這種假值。真正的設定值該用環境變數存，`.env` 本身列進 `.gitignore`、永遠不進版控。更好的做法是讓 Claude 用它自己的工具去讀檔案，而不是你手動貼內容——差別在於：讀檔案至少可以用權限規則擋下來（見下面），貼進對話框就沒有回頭路了。

### 情境二：把 API key 寫進 CLAUDE.md，方便 Claude 知道怎麼連線，結果 push 上 public repo

你想讓 Claude 知道怎麼呼叫某個 API，圖方便直接把金鑰打進 `CLAUDE.md`。心想反正是自己專案，應該沒差。結果這個 repo 是 public 的，一 push 上去，任何人在 GitHub 上都能直接看到這把鑰匙。

**風險**：`CLAUDE.md` 的內容會被整份讀進每次對話的 context——這本身沒問題，但也代表它從來就不是拿來存密鑰的地方，它是「說明文件」，不是「保險箱」。public repo 一 push，鑰匙就是全世界都看得到，跟把鑰匙貼在自家大門上沒兩樣。

**該怎麼做**：`CLAUDE.md` 只寫「怎麼呼叫這個 API」這種說明性文字，金鑰本身一律用環境變數存。官方本身也是把使用者的 API key／token 存進 macOS 的 Keychain、在 Windows 與 Linux 上用檔案權限保護——這提示了一個原則：金鑰該交給「有權限保護機制」的地方，不是明碼寫進你會到處傳閱、甚至公開的文件裡。

<Card title="已經 push 上去了怎麼辦" icon="warning">
別只是把那行刪掉重新 commit——git 歷史紀錄還留著，遠端 repo 也可能已經被抓取過。金鑰一旦公開過，唯一正確的處理是**立刻到服務商那邊撤銷、重新產生一把新的**，把舊金鑰當作已經外流來對待。
</Card>

### 情境三：把身分證字號、病歷、信用卡號貼給 Claude 問「幫我整理」

你想讓 Claude 幫忙整理報稅資料或家人的健康紀錄，直接把身分證字號、信用卡卡號、診斷紀錄整段貼上去。

**風險**：這些內容都會經過網路送到 Anthropic 的伺服器處理。官方對資料保留期限、存取範圍都有規範，但「有隱私保護機制」不等於「什麼都可以傳」——能不傳就不傳，尤其這類個資（身分證字號、病歷）不像密碼，外流了沒辦法「換一個」。

**該怎麼做**：先去識別化，把姓名、身分證字號、卡號換成「王小明」「卡號末四碼 XXXX」這種假資料，只留 Claude 真的需要幫忙判斷的部分——例如「這筆支出該歸在哪一類」根本不需要卡號本身。想深入了解資料實際會流向哪裡，看 [資料到底傳去哪裡](../../trust/where-does-data-go/)。

### 情境四：把公司內部機密文件或客戶名單整份丟給 Claude 做摘要

你想偷懶，把整份公司內部策略文件或客戶聯絡清單丟給 Claude 摘要重點。

**風險**：這些內容不是你一個人的。牽涉公司規範，或者是別人的隱私——客戶資料尤其屬於「別人的個資」，不是你可以單方面決定要不要傳出去的東西。

**該怎麼做**：先確認公司或組織有沒有明訂「能不能用 AI 工具處理這類資料」的規定，沒有明訂就先問，別自己擅自決定。確定不行的資料類別，連摘要都不要貼。

## ✅ 萬用檢查法：貼之前先問自己一句

不用背清單，記一句話就夠：

**這段東西如果被別人看到，會怎樣？**

答得出「沒差」，就貼。答案是「會出事」，先停下來，換成假資料，或乾脆不貼。

## 🔒 用設定擋住：讓 Claude Code 自己都讀不到

上面講的是「你自己別貼」。還有一層更保險的做法：**直接設定讓 Claude Code 連讀取的機會都沒有**。

在 `.claude/settings.json` 加上這樣的規則：

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
```

這是官方文件裡的機制，不是自己土法煉鋼：`deny` 規則生效後，符合條件的檔案 Claude Code 連嘗試讀取都會直接被擋下來，不是「問你要不要讀」而已。想知道 `allow`／`deny`／`ask` 怎麼互相搭配、規則比對順序是什麼，看 [settings 與權限模式](../../reference/settings-permissions/) 有完整說明。

:::caution
這條規則擋的是 Claude Code 這個工具本身，跟你自己手動貼進對話框是兩回事——設定擋得再好，你自己複製貼上還是繞得過去。兩件事都要顧到。
:::

**你會看到**：Claude 列出檔名（如果有的話），不會主動唸出裡面的內容。如果資料夾裡本來就沒有這類檔案，那也是好結果——代表這個資料夾目前是乾淨的。

<details>
<summary>做壞了怎麼辦：已經不小心貼出去、或已經 commit 進去了</summary>

- **貼進對話框了**：如果是密碼或 API key，第一件事是去撤銷、重新產生一把新的，不是刪掉那則訊息就沒事——訊息可能已經被記錄下來。
- **已經 commit、甚至 push 上去了**：光刪檔案、改一行重新 commit 沒有用，git 歷史紀錄還留著原本的內容。正確處理是撤銷舊金鑰、換一把新的；想知道 git 版本紀錄本身能不能當「復原」用，看 [git 是你的復原鍵](../../trust/git-as-undo/)。
- 不確定自己貼過什麼敏感資料出去？把這件事當成「已經外流」處理最安全，換一把新金鑰的成本，遠低於猜錯的代價。

</details>

## 🔑 該進你的 CLAUDE.md 嗎

可以。把你這個專案「絕對不能給 Claude 看的檔案類型」寫進 `permissions.deny`（不是寫進 `CLAUDE.md`——這點上面情境二講過，`CLAUDE.md` 不是保險箱）。

如果你想在 `CLAUDE.md` 留個提醒，可以寫類似這樣一條：「涉及金鑰、個資的檔案一律先問過我，不要主動讀取或摘要」——這是「行為邊界」類規則，跟 [打造你自己的 CLAUDE.md](../../learn/your-own-claude-md/) 那課教的起步心法完全相同。
