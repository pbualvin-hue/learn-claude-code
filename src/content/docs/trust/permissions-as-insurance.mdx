## 🛡️ 為什麼會一直跳出來問你

Claude 想改檔案、跑指令、或連去網路上某個地方之前，預設會先停下來，跳一個小視窗問你「這樣可以嗎？」。

這不是它笨手笨腳、也不是壞掉。這是故意設計的關卡——就像新來的助理，重要的事情動手前先跟你確認一聲，而不是自己悶頭做。

這道關卡就是你的保險。保險平常感覺不到它的存在，但真的出狀況的時候（例如它想刪的東西其實你不想刪），就是這一下攔住了。

## ✋ 提示跳出來時，代表什麼

看到提示，代表 Claude 想做一件「還沒被你允許過」的事——可能是編輯某個檔案、跑一行指令、或呼叫某個外部服務。

它會列出想做的動作，等你選允許或拒絕。你選了，它才會真的動手；你不選，它就停在那裡等你。

換句話說：只要你還沒按下允許，什麼事都不會發生。這是你可以慢慢看的地方，不用怕耽誤它。

## 🎚️ 四種模式，白話版

技術上 Claude Code 有好幾種「權限模式」，決定它多常跳出來問你。新手大概記住這四種常見的就夠：

<CardGrid>
	<Card title="一般模式：逐次問你" icon="approve-check">
		預設就是這個。每一種新的動作，它第一次做之前都會先問你。最安全、也最適合剛開始用的時候。
	</Card>
	<Card title="自動接受編輯" icon="setting">
		檔案編輯、還有 `mkdir`／`mv`／`cp` 這類常見的檔案操作，它會自動做、不再逐次問。適合你已經在看著 `git diff` 確認結果的情況。
	</Card>
	<Card title="只讀規劃，不動手" icon="information">
		Claude 只會讀檔案、跑一些唯讀指令做調查，寫出一份計畫給你看，但不會真的改你的東西。適合先讓它摸清楚狀況再決定要不要動手。
	</Card>
	<Card title="全部放行（風險最高）" icon="warning">
		跳過幾乎所有提示，Claude 想做什麼就直接做。等於把保險拆了，往下看風險說明。
	</Card>
</CardGrid>

想知道每種模式技術上怎麼設定、怎麼切換，看 [settings 與權限模式](../../reference/settings-permissions/)。這一頁只講一件事：怎麼安心地判斷。

## 🔍 看到提示時，怎麼判斷

<Steps>

1. **先看清楚它要做什麼**：是改哪個檔案、跑哪一行指令，還是要連去哪個網址。提示上通常會寫出來，花三秒看一眼。

2. **問自己：這跟我剛剛講的事有關係嗎**。如果你叫它「整理這個資料夾」，結果它想跑一個看起來在對外送資料的指令，這就對不起來，該停下來多問一句。

3. **看不懂就先不要按允許**。可以直接打字問 Claude「你這個指令是要做什麼？」，等它解釋清楚你懂了，再決定。看不懂還是按下去，才是真正的風險。

4. **這個資料夾／專案，你信任來源嗎**。如果是隨手抓來路不明的專案，多一分警覺；自己從頭寫的專案，通常可以放心一點。

</Steps>

不要無腦一直按允許——尤其是一連串提示跳出來的時候，很容易養成「看到就按」的反射動作。真正需要你判斷的時刻，往往就藏在那一長串裡面的某一個。

:::caution[全部放行的風險]
「全部放行」模式官方自己講得很白：它不提供任何防護，擋不住惡意內容誘導 Claude 做壞事（術語叫 prompt injection），也擋不住 Claude 自己判斷錯誤。

等於把這一整頁講的保險機制整個拆掉。官方建議只在容器、虛擬機這種「就算壞掉也沒差」的隔離環境用。一般在自己電腦上開發，不要開這個模式。
:::

## 🪝 一個誠實的但書：hooks 不會再問你

權限提示不是萬能防護——如果你（或別人）設定了 [hooks](../../reference/hooks/)，情況會不一樣。

hooks 是「設定好之後，條件符合就自動執行」的機制，它不會像一般動作那樣每次跳出來問你。這是特意設計的：hooks 本來就是要讓某些事不用每次都手動確認。

代價是：如果裝進來的 hook 藏著壞心的東西，你不會有機會攔下它。所以只裝你自己寫的、或來自你信任的來源的 hook——尤其是打開一個不熟悉的專案時，它可能連帶著別人寫好的 hook 一起進來，這點常被忽略。

## 📝 這一課有沒有該記進 CLAUDE.md 的？

如果你發現自己對某一類提示（例如某個常跑的指令）每次都選允許，可以考慮把它寫成 [設定裡的允許規則](../../reference/settings-permissions/)，而不是每次手動按。但這是「省麻煩」用的機制，不是「看不懂也直接放行」的藉口——這兩件事不一樣。
