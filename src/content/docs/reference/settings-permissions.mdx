把 Claude Code 想成一位很能幹的新助理：剛開始共事時，你會希望他做每一件事之前都先問你一聲「這個可以做嗎？」；熟悉之後，你可能放心讓他自己處理例行公事（例如整理檔案），但重要的事（例如刪東西、對外送出請求）還是想先審過。`settings.json` 加上「權限規則」，就是你設定「哪些事可以自動做、哪些事一定要先問我」的地方。

上一頁提到 [`CLAUDE.md`](../claude-md-memory/) 只是「建議」，Claude 會盡量照做但不保證。權限規則不一樣——它是 Claude Code 這個工具本身在執行層擋下來的關卡，不管 Claude 想不想做，符合擋下條件就是擋下來，這也是它被稱作「安全保險」的原因。

## 進一步了解

### settings.json 放在哪裡

| 範圍 | 位置 | 誰會用到 |
| --- | --- | --- |
| 使用者層 | `~/.claude/settings.json` | 你個人在所有專案的設定 |
| 專案層 | `.claude/settings.json` | 團隊共用，進版控 |
| 本機層 | `.claude/settings.local.json` | 只有你自己的專案內覆寫，不進版控 |
| 管理層 | 由組織 IT 統一部署 | 組織政策，個人設定無法覆蓋 |

多份設定同時存在時，優先順序是：管理層 > 命令列參數 > 本機層 > 專案層 > 使用者層。但**權限規則是例外**：來自不同層級的 allow / deny / ask 規則會合併在一起套用，不是後面蓋掉前面。

### 權限規則的語法

`settings.json` 裡的 `permissions` 欄位用三個陣列控制：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)"
    ]
  }
}
```

- `allow`：符合的動作自動放行，不再詢問
- `deny`：符合的動作直接擋下，Claude 連嘗試的機會都沒有
- `ask`：符合的動作每次都跳出來問你

規則比對順序固定是 **deny 先、ask 次、allow 最後**，第一個比對到的規則就決定結果，跟規則寫得多精細無關。這是常見誤解：很多人以為「寫得越具體的規則優先」，但一條寬鬆的 `Bash(aws *)` deny 規則，會擋住所有符合的呼叫，即使同時有一條更精細的 `Bash(aws s3 ls)` allow 規則也一樣。

### 權限模式

除了逐條規則，Claude Code 還有幾種整體「模式」，可以用 `defaultMode` 設定，或在對話中切換：

| 模式 | 行為 |
| --- | --- |
| `default`（介面上顯示為 Manual） | 標準行為：每種工具第一次使用都先問你 |
| `acceptEdits` | 自動接受檔案編輯，以及 `mkdir`／`mv`／`cp` 這類常見檔案操作 |
| `plan` | Claude 只讀檔案、跑唯讀指令做調查，不會真的動你的原始碼 |
| `auto` | 幾乎不跳提示，但每個非唯讀動作先經過一個**獨立的安全分類器模型**審查——越權、可疑、破壞性的動作會被擋下 |
| `dontAsk` | 反向邏輯：凡是本來要問你的動作一律**自動拒絕**，只有 allow 清單預先核准的工具能執行——給 CI 或鎖死環境用 |
| `bypassPermissions` | 幾乎跳過所有提示（少數如刪除根目錄/家目錄的指令仍會攔） |

幾個模式的補充說明：

- **`auto` vs `bypassPermissions` 一句話**：bypass 是「全部放行、沒人看著」；auto 是「放行前有另一個模型在看」。想少被打擾，官方建議優先考慮 auto，而不是 bypass。
- `auto` 有可用條件（方案、模型、平台都有要求，詳見官方 permission-modes 頁）：模式選單裡沒看到它，通常是條件不符，不是壞掉。
- 防守設計：`defaultMode: "auto"` 寫在專案層 `.claude/settings.json` 或 `settings.local.json` 會被**忽略**——repo 不能替自己打開 auto 模式；要開只能寫在你自己的 `~/.claude/settings.json`。
- `dontAsk` 模式下，`ask` 規則命中的動作會被直接拒絕而不是詢問（這個模式從頭到尾不會收集任何回答）。
- `bypassPermissions` 風險最高：官方明確建議只在容器或 VM 這類「壞了也沒關係」的隔離環境用，一般開發專案不建議開啟。

## 🛠️ 實戰進階

### 一份可抄的窄允許清單

{/* 範例已去識別化：路徑與規則皆為通用示意，非任何真實專案設定 */}

```json
{
  "permissions": {
    "allow": [
      "Bash(git status*)",
      "Bash(git log*)",
      "Bash(git diff*)",
      "Edit(docs/**)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./secrets/**)"
    ]
  }
}
```

- `Bash(git status*)` / `log` / `diff`：這三個其實本來就在官方內建的「唯讀指令」清單裡，任何模式下都不會跳出詢問，寫進 `allow` 主要是「白紙黑字讓團隊看得到意圖」，不是真的靠它才放行——這點常被誤會成「有寫才有用」。
- `Edit(docs/**)`：寫入規則只認 `Edit(路徑)` 這一種形態（詳見下方「限制與注意」），`Write(docs/**)` 或 `Glob(docs/**)` 這類寫法看起來合理，實際上啟動時只會收到警告、完全不生效。
- `Read(./.env)` 擋最常見的密鑰外洩管道；但 deny `Read` 只會連帶擋住同路徑的 `Edit`（含新建檔案），擋不住 `Write` 直接蓋一個新的 `.env`，真的要滴水不漏得再補一條 `Edit(./.env)` deny。
- `Read(./secrets/**)`：沿用 gitignore 語法，擋的是「目前目錄以下」的 `secrets/`，不是整台機器——想擋全機器要寫成 `Read(//**/secrets/**)`。

### 前綴比對的實務坑：旗標一變就失守

下方「限制與注意」提到用萬用字元限制 curl 網址很脆弱（多個旗標、換協定都能繞過），這裡補一個更貼身的版本：`allow` 對 Bash 指令是「字串前綴＋詞界」比對，同一個指令換個旗標組合，在規則眼裡就是完全不同的字串。

**本站實測**：本站排程管線抓官方 changelog 時，只寫一條「乾淨版」`Bash(curl https://raw.githubusercontent.com/...*)` 規則，結果草稿有時候多帶 `-s`（安靜模式）、有時候多帶 `-sL`（安靜＋跟隨轉址）抓同一個網址，兩種都各自被卡下來重問一次——旗標插在網址前面，前綴就對不上了。最後解法不是把規則寫得更「聰明」，而是老實地對同一個網址前綴分別列出「無旗標／`-s`／`-sL`」三條規則，才把常見的呼叫方式都涵蓋到。真的要一勞永逸，官方建議的做法是別再用 Bash 規則圍堵網址，改成 deny 擋掉 `curl`／`wget` 整個指令，另外用以網域為單位的 `WebFetch(domain:example.com)` 放行。

### 分層怎麼選

| 放這層 | 位置 | 放什麼 | 例子 |
| --- | --- | --- | --- |
| 專案層 | `.claude/settings.json` | 團隊都該遵守、進版控的標準規則 | 團隊共識「禁止直接 `git push` 到 main」，寫成 `deny: ["Bash(git push origin main)"]` |
| 本機層 | `.claude/settings.local.json` | 只有你自己會用到、不想讓隊友也套用的例外 | 你電腦上一支本機除錯腳本的路徑允許，或實驗階段暫時放寬的規則 |
| 使用者層 | `~/.claude/settings.json` | 你個人在所有專案都想要的習慣 | 允許讀你自己的 `~/.zshrc`、或設定你偏好的 `defaultMode` |

分層要問的其實是「這條規則該不該進版控、給不給隊友看到」，不是「哪一層優先」——如同前面提過的，權限規則是跨層合併，不是後面蓋掉前面。

### 跟相鄰機制的邊界

- **權限規則（本頁）**：工具層硬擋，不管 Claude 想不想做，符合條件就是擋下來——確定「無論如何都不能做」時用它。
- **[hooks](../hooks/)**：想在某個動作前後插入你自己寫的檢查邏輯（不是單純允許/擋，而是要跑一段程式去判斷）時用。
- **[CLAUDE.md 與記憶](../claude-md-memory/)**：模型層的建議，只是讓 Claude「傾向」怎麼做，沒有強制力；想再理解為什麼權限規則比建議可靠，可看 [權限是你的保險](../../trust/permissions-as-insurance/)。

## 限制與注意

- **帶路徑的規則只有 `Edit(路徑)` 與 `Read(路徑)` 兩種寫法**：官方文件明載「`Edit` 規則適用於所有會編輯檔案的內建工具」（含 Write）。自 v2.1.210 起，寫 `Write(路徑)`、`NotebookEdit(路徑)`、`Glob(路徑)` 這類規則會在啟動時收到警告——這些寫法的比對方式不同，官方指示一律改用 `Edit(路徑)`／`Read(路徑)`。
- **權限規則擋的是 Claude Code 這個工具，不是 Claude 這個模型**：`CLAUDE.md` 裡寫「不要跑 `rm -rf`」只是建議，Claude 有可能沒照做；但 `permissions.deny` 裡寫的規則，是 Claude Code 直接不執行，兩者不能互相替代。
- 用萬用字元限制 Bash 指令的參數其實很脆弱。官方文件直接點名這個坑：例如想寫一條規則只允許 curl 抓 GitHub 網址，`Bash(curl http://github.com/ *)` 這種寫法看似安全，實際上加個參數（`curl -X GET http://github.com/...`）、換個協定（`https://`）、或透過會轉址的短網址，都能繞過去。真的要限制對外連線，建議改用 deny 擋掉 `curl`／`wget`，改用 `WebFetch(domain:example.com)` 這種以網域為單位的規則。
- 專案層 `.claude/settings.json` 裡的 `allow` 規則，要等你手動通過「workspace trust」的信任對話框才會真的生效——不是寫進檔案就立刻有效，這點常被誤會。`deny` 和 `ask` 規則不受此限制，因為它們只會限制、不會放行。
- `bypassPermissions` 模式仍然會攔下針對根目錄或家目錄的刪除指令（例如 `rm -rf /`），但這是最後一道防線，不是常態安全機制，不要把它當成「開了也沒差」。
