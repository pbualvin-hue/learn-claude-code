:::note[先備知識]
先看 [settings 與權限模式](../settings-permissions/) 與 [CLAUDE.md 與記憶](../claude-md-memory/)，這頁把它們放在同一個資料夾結構下看。
:::

`.claude` 資料夾就像這個專案的「個人設定抽屜」：裡面放的是「Claude 在這個專案該遵守什麼規則」「哪些操作不用每次都問」
「有沒有自訂的捷徑指令」這類設定。專案的抽屜大家共用（進版控、team 一起看），你自己電腦裡還有一個個人抽屜
（`~/.claude`），放的是不管開哪個專案都要套用的個人習慣。

## 進一步了解

### 專案層：`.claude/`（放在專案根目錄）

| 路徑 | 放什麼 | 進版控？ |
| --- | --- | --- |
| `CLAUDE.md`（或 `.claude/CLAUDE.md`） | 團隊共用的專案指令：架構、慣例、常用指令 | 是 |
| `CLAUDE.local.md` | 你個人的專案偏好（測試資料、本機網址等） | 否，加進 `.gitignore` |
| `.claude/settings.json` | 團隊共用設定：權限規則、環境變數等 | 是 |
| `.claude/settings.local.json` | 你個人的設定覆寫 | 否 |
| `.claude/rules/*.md` | 依檔案路徑條件載入的細部規則（如只在改 `*.ts` 時載入） | 是 |
| `.claude/skills/<name>/SKILL.md` | 專案技能，會產生 `/<name>` 指令 | 是 |
| `.claude/commands/<name>.md` | 舊版自訂指令格式，效果等同 skill，目前仍可運作 | 是 |
| `.claude/agents/*.md` | 專案專屬的 subagent 設定 | 是 |
| `.claude/output-styles/` | 輸出風格設定 | 是（⚠ 未驗證：官方文件僅提及此路徑存在，未詳述內部檔案格式） |

### 使用者層：`~/.claude/`（你的家目錄下）

套用到你電腦上**所有專案**，不會跟團隊共用：

| 路徑 | 放什麼 |
| --- | --- |
| `~/.claude/CLAUDE.md` | 你的個人全域指令（程式風格偏好、個人常用工具等） |
| `~/.claude/settings.json` | 個人層級設定，優先度低於專案設定 |
| `~/.claude/skills/`、`~/.claude/agents/`、`~/.claude/rules/`、`~/.claude/output-styles/` | 上述各項目的個人版本，任何專案都能用 |
| `~/.claude/projects/<project>/memory/` | auto memory 存放處：Claude 自己寫的筆記，含 `MEMORY.md` 索引檔與其他主題檔 |

`~/.claude/projects/<project>/memory/` 比較特別：不是你手寫的規則，是 Claude 在對話中自己判斷「這個以後可能有用」而寫下的筆記，
機器本地儲存，同一個 git repo 的所有 worktree 共用一份，但不會同步到別台電腦。

### 兩層怎麼疊加

同名設定衝突時，範圍大的先讀、範圍小的後讀，越晚讀進來的優先度越高。實際完整順序是：

**Managed（組織層）＞ 命令列參數 ＞ Local（`settings.local.json`）＞ Project（`settings.json`）＞ User（`~/.claude/settings.json`）**

容易忽略的一點：本機的 `.claude/settings.local.json` 優先度比 commit 進 repo、team 共用的 `.claude/settings.json` **更高**——不是單純「專案層蓋過使用者層」這麼簡單，同一台機器上，你自己的 local 覆寫一樣會蓋過整個團隊共用的專案設定。
Managed（如 macOS 的 `/Library/Application Support/ClaudeCode/`）由 IT／DevOps 集中管理、一般使用者無法覆蓋，多半只在公司統一發放電腦時才會遇到，這裡不展開。

## 🛠️ 實戰進階

### 範例一：一個實務上完整的專案 `.claude/` 目錄＋`.gitignore`

把 `.claude/` 目錄當成「這個專案的協作契約」來設計時，實務上長這樣：

```text
my-project/
├── .claude/
│   ├── CLAUDE.md              # 團隊共用指令（進版控）
│   ├── settings.json           # 團隊共用權限、hooks（進版控）
│   ├── settings.local.json     # 你個人的覆寫（不進版控）
│   ├── rules/
│   │   ├── code-style.md       # 進版控
│   │   └── testing.md          # 進版控
│   ├── skills/
│   │   └── deploy/SKILL.md     # 進版控
│   └── agents/
│       └── reviewer.md         # 進版控
├── CLAUDE.local.md              # 你個人的專案筆記（不進版控）
└── .gitignore
```

```text title=".gitignore"
.claude/settings.local.json
CLAUDE.local.md
```

**注意這條 `.gitignore` 不是自動幫你寫好的**：Claude Code 自己建立 `.claude/settings.local.json` 時，會順手幫你在 git 設定裡加忽略；但如果是你自己手動建立這個檔案（或是 `CLAUDE.local.md`），就不會自動排除，得自己加進 `.gitignore`，否則個人設定或私人筆記會被一起 commit 進團隊 repo。

新成員加入專案時，建議照這個順序讀：先看 `.claude/CLAUDE.md`（團隊指令）、再看 `.claude/settings.json`（權限規則跟 hooks，了解哪些操作被自動允許或擋下）、有 `.claude/rules/` 的話挑跟自己要改的檔案類型相關的看。`.claude/settings.local.json` 跟 `CLAUDE.local.md` 是別人的個人設定，不需要看、也不該去改。

### 範例二：用 `/status`、`/context`、`/doctor` 確認某條設定實際從哪一層生效

多層設定疊加最常見的困惑是「明明 `.claude/settings.json` 裡設了 deny 規則，怎麼在我這裡好像沒生效」。排查順序：

1. **`/status`**：列出這次 session 實際載入的設定來源（`Setting sources`），每個來源必須「至少含一項設定且成功解析」才會出現在清單裡——如果某個 settings 檔案 JSON 語法寫錯，它會直接消失在清單外，而不是報錯給你看。
2. **`/doctor`**：檢查所有設定檔的 JSON 是否合法，抓出寫錯的地方。
3. 確認完檔案有載入、JSON 也沒錯之後，若行為還是跟預期不同，檢查 **`.claude/settings.local.json` 裡有沒有衝突的規則**——多層優先順序是 Managed ＞ 命令列參數 ＞ **Local（`settings.local.json`）＞ Project（`settings.json`）＞ User（`~/.claude/settings.json`）**，local 的優先度比 project 高，一條允許規則寫在你自己的 `settings.local.json` 裡，會直接蓋過團隊寫在 `settings.json` 裡的擋下規則。

CLAUDE.md 沒被遵守時排查方式類似，但指令不同：跑 **`/context`** 確認這次 session 實際載入了哪些 CLAUDE.md／`CLAUDE.local.md`（載入清單裡沒看到，代表 Claude 根本沒讀到這份指令，不是「讀到但沒照做」），跑 **`/memory`** 直接開啟並編輯這些檔案。

### 跟相鄰機制的邊界

- **這一頁**：`.claude/` 目錄結構本身——哪個檔案該放在哪、進不進版控、多層（managed／local／project／user）疊加時誰蓋過誰，是團隊協作時「誰的設定會生效」的總覽入口。
- **[settings 與權限模式](../settings-permissions/)**：`settings.json` 內部欄位怎麼寫、權限規則（allow／deny／ask）的語法細節——這頁只講檔案放哪裡，欄位內容看那頁。
- **[CLAUDE.md 與記憶](../claude-md-memory/)**：`CLAUDE.md` 該寫什麼內容、auto memory 怎麼運作——這頁只講 `CLAUDE.md` 放在哪一層、疊加順序，內容怎麼寫看那頁。
- **[slash commands 總覽](../slash-commands/)**：`.claude/commands/`、`.claude/skills/` 實際會變成什麼指令。
- **[hooks](../hooks/)**：`.claude/settings.json` 裡的 hooks 欄位具體怎麼寫。

### 常見誤用

- **以為寫進 `.claude/settings.json` 就一定對所有人生效**：容易忽略 `.claude/settings.local.json` 的優先度比專案層 `settings.json` 高——本機的 local 覆寫會蓋過 commit 進 repo 的團隊設定，出現「同事說這條規則生效了，我這邊卻沒有」的落差時，先檢查自己有沒有 local 覆寫，而不是懷疑團隊設定本身寫錯。
- **手動建立 `settings.local.json` 或 `CLAUDE.local.md` 卻忘了排除進版控**：只有 Claude Code 自己建立這個檔案時才會自動幫你設定 git 忽略；自己手動建立的話不會自動排除，一沒注意就把個人偏好或私人筆記 commit 進團隊共用的 repo。

## 限制與注意

- **`.claude/commands/` 與 `.claude/skills/` 同名時 skill 優先**：兩者效果相同（都會產生 `/name` 指令），但同時存在時 Claude Code 會用 skill 版本，不是報錯，也不是兩者都跑。
- **`output-styles/` 的內部格式本頁未驗證**：官方文件確認這個路徑存在（在說明「使用者層可以放什麼」時提到），但沒有進一步文件描述檔案該怎麼寫，本站目前也還沒找到更詳細的官方頁面，先標記為 ⚠ 未驗證。
- **`.claude/settings.local.json` 不一定完全免審查**：它通常是「你自己的檔案」不用經過 workspace trust 確認，但如果這個檔案被 commit 進 repo、或 `.claude` 本身是 symlink，就會被當成「repo 可能提供的檔案」，一樣要通過信任確認才會套用。
- **專案 `.claude/settings.json` 的 allow 規則不是加了就馬上生效**：第一次在某個資料夾用 Claude Code 時，要先通過畫面上的信任確認（trust dialog），allow 規則跟 `additionalDirectories` 才會真的套用。
- **這個結構會隨版本演進**：例如「custom commands 併入 skills」是相對晚近的變動，本頁以 base_version 2.1.207 為準，未來版本的目錄結構可能再變。
