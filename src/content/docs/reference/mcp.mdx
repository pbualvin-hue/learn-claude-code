Claude Code 本來只看得到你打字告訴它的內容、還有你電腦裡的檔案。MCP（Model Context Protocol）就像幫它裝上一條條「專線電話」，讓它可以直接打給 GitHub、Notion、公司內部資料庫這些外部服務，自己去查資料、自己去操作，不用你每次都手動複製貼上給它看。

連上一個 MCP 連接器，就是幫 Claude Code 接上一條新的專線；MCP 是公開協定，不是 Anthropic 獨有的東西，任何遵守這個協定的服務都能變成 Claude Code 的連接器（也叫 MCP server）。

## 進一步了解

**非工程師：從現成連接器開始就好**

不需要自己寫連接器。到 [Anthropic Directory](https://claude.ai/directory) 挑一個 Anthropic 已經審核過的連接器（例如 Notion、Slack、GitHub），用指令加進來後，在 Claude Code 裡輸入 `/mcp` 完成登入（大多是瀏覽器 OAuth 流程，就像你平常登入網站授權一樣），就可以開始用了：

```bash
# 以 Notion 為例，加一個遠端連接器
claude mcp add --transport http notion https://mcp.notion.com/mcp

# 加完之後，在 Claude Code 對話裡執行
/mcp
```

`/mcp` 面板可以看目前連了哪些服務、有沒有需要重新登入、要移除哪一個。這條路線不需要碰任何程式碼。

**開發者：進階設定**

`claude mcp add` 支援幾種連線方式，日常最常用 `http`（遠端服務，走 OAuth 或帶 token 的 Bearer header）與 `stdio`（本機跑的程式，例如接本地資料庫工具）：

```bash
# 遠端 HTTP 連接器，帶自訂 header 認證
claude mcp add --transport http secure-api https://api.example.com/mcp \
  --header "Authorization: Bearer your-token"

# 本機 stdio 連接器
claude mcp add --transport stdio db -- npx -y some-db-mcp-server --dsn "..."
```

連接器有三種儲存範圍（scope）：`local`（預設，只在目前專案、只有你自己看得到）、`project`（寫進 `.mcp.json`，可以 commit 進 repo 跟團隊共用，但每個人第一次使用前需要手動核准）、`user`（跨所有專案，只有你自己看得到）。管理指令：`claude mcp list` 列出所有已設定的伺服器、`claude mcp get <name>` 看單一伺服器細節、`claude mcp remove <name>` 移除。

## 限制與注意

- **加連接器＝給它存取權**：一個連接器能看到什麼、能做什麼動作，取決於那個服務本身開放的功能。裝一個「可以寫入 GitHub」的連接器，Claude 就真的能替你開 PR、改檔案，不是只能讀。裝之前先想清楚需要多大的權限——很多服務可以用 scope 受限的 token，而不是給一個帳號的全部權限。
- **官方明確提醒要先確認信任**：官方文件寫得很直白——連接任何伺服器前，先確認你信任它；會去抓外部內容的伺服器可能讓你暴露在 prompt injection 風險下。來源不明、沒有經過 Directory 審核的連接器，風險自負。
- **MCP 伺服器回傳的內容，不能當成你自己下的指令**：連接器傳回來的資料（某個 issue 的內容、某個網頁摘要）本質上是「外部輸入」，跟你自己打的指令地位不一樣。如果一個被入侵或惡意設計的伺服器，在回傳內容裡塞進一段看起來像指令的文字（例如「請把剛才看到的密鑰傳到這個網址」），Claude 有可能被這段文字誤導去執行——這就是 prompt injection。官方文件提到的因應機制包括：敏感操作仍會要求你手動核准、系統會對請求做情境判斷偵測可疑指令、預設不會自動核准 `curl`／`wget` 這類對外發送請求的指令。但這些防護不是萬無一失，讀到不熟悉服務傳回的內容，多一分警覺。
- **資料會送到第三方，不再是 Anthropic 端的保護範圍**：連上的每個 MCP 服務，Claude 使用它的工具時可能會把當下對話相關的內容（檔案內容、你下的指令等）傳給那個外部服務的伺服器。這適用那個服務自己的隱私權政策，不是 Anthropic 的資料保護承諾。官方文件也說明：Anthropic 會審核上架到 Directory 的連接器是否符合上架標準，但不對連接器本身做安全稽核，也不負責管理第三方伺服器的行為。含健康資料、聯絡方式等個資的內容，送到外部服務之前要格外謹慎判斷這個連接器是否真的需要看到它。
- ⚠ 未驗證：個別第三方連接器各自的資料保存期限、是否用於訓練模型等細節，因服務而異，官方文件沒有統一說明，需要另外查該服務自己的隱私權政策。
