想像你請 Claude Code 幫你查一個 bug，過程需要翻十份 log 檔案。如果這十份 log 全部攤開在你眼前的對話裡，你的「桌面」（也就是 Claude 的對話記憶空間，術語叫 context）很快就被雜訊塞滿——真正該討論的重點反而被淹沒。

subagent 就像請一位助理到隔壁房間去翻資料，翻完只帶一句結論回來給你，原始的十份 log 留在助理房間裡，不會佔用你的桌面。這位助理有自己的一套工作方式、只能碰特定的工具，做完事就回報結果，你的主對話畫面乾乾淨淨。

## 進一步了解

用術語講：subagent 是一個獨立跑的 Claude Code agent，有自己的 context window（獨立記憶空間）、自己的 system prompt、可用工具清單與權限設定。當你的請求符合某個 subagent 的 `description`，Claude 就會把任務委派過去，subagent 獨立工作完再把結果回傳給主對話。

**內建 subagent**（不用自己設定，Claude 會視情況自動使用）：
- `Explore`：唯讀，專門搜尋、理解程式碼庫，不能改檔案
- `Plan`：plan mode 底下負責先做研究的唯讀 agent
- `general-purpose`：可讀寫，處理需要探索＋動手改的複雜任務

**自訂 subagent** 是一個 Markdown 檔加上 YAML frontmatter，放在：

| 位置 | 適用範圍 |
| --- | --- |
| `.claude/agents/` | 只在這個專案 |
| `~/.claude/agents/` | 你所有專案都能用 |

最快的做法是直接請 Claude 幫你寫，例如：

```text
Create a personal code-improver subagent in ~/.claude/agents/ that scans
files and suggests improvements. Make it read-only and use Sonnet.
```

產生出來的檔案長這樣：

```markdown
---
name: code-improver
description: Scans files and suggests improvements for readability, performance, and best practices. Use after writing or modifying code.
tools: Read, Grep, Glob
model: sonnet
---

You are a code improvement specialist. For each issue you find, explain
the problem, show the current code, and provide an improved version.
```

frontmatter 只有 `name` 跟 `description` 是必填，`tools`（工具白名單）、`model`（跑哪個模型）等都可以省略沿用預設。

**叫用方式**有三種，力道由弱到強：直接在對話裡提到 subagent 名字（Claude 自行判斷要不要用）、用 `@subagent 名字` 明確指定這次一定要用它、或用 `--agent <name>` 讓整個 session 都以這個 subagent 的身分運作。

**何時該用**：
- 側任務會產生大量你不需要留在主對話的輸出——log、搜尋結果、測試輸出
- 想限制某個任務能碰的工具，例如做 code review 時只給唯讀權限
- 想平行調查多個獨立範圍，例如同時研究認證模組、資料庫模組、API 模組，各自在自己的 context 裡工作，最後由 Claude 彙整結果

## 🛠️ 實戰進階

### 一個可抄的自訂 subagent 定義

自訂 subagent 是「一個 Markdown 檔＋YAML frontmatter」，存在 `.claude/agents/`（只在這個專案）或 `~/.claude/agents/`（全專案通用）。以一個「文件摘要員」為例——教材草稿常需要先有人讀懂一大份官方文件再回報重點，很適合外包：

```markdown
---
name: doc-summarizer
description: Reads a long document or spec and returns a concise summary with key points and open questions. Use when asked to digest a long file, spec, or changelog.
tools: Read, Grep, Glob
model: sonnet
---

You are a documentation summarizer. Read the requested file(s) end to
end, then report: (1) a 3-5 bullet summary, (2) any terms that need
clarification, (3) anything that looks outdated or contradictory.
```

frontmatter 只有 `name`、`description` 必填；`tools`（工具白名單）、`model`（跑哪個模型）都可省略，沿用預設。

### description 決定何時被委派

主對話收到你的請求後，會拿「請求本身」跟每個 subagent 的 `description` 比對，判斷這次該不該交出去。官方講法是委派依據三者：這次的任務描述、subagent 的 `description` 欄位、目前的 context。所以 description 要寫「什麼情境用我」，不是「我是什麼」——像上面例子寫「Use when asked to digest a long file...」，而不是只寫「Summarizes documents」。想讓 Claude 更積極主動委派，官方建議在 description 裡加類似 "use proactively" 的字眼。

### context 隔離是本體

subagent 一啟動就是全新、空白的 context window：看不到主對話歷史、你已讀過的檔案、你們剛聊過什麼。這正是它的價值所在——翻十份 log、讀一份幾萬字的 spec，這類會塞爆主對話的粗活丟給它，做完只帶結論回來，你的主對話依然乾淨。

也因為這樣，常見誤用是「以為 subagent 知道你們剛聊過的事」——它不知道，Claude 只會幫你寫一段委派訊息交代任務，細節不會自動帶過去。委派時要把必要背景寫進任務本身，例如與其說「繼續剛才那個」，要說「讀 `docs/spec.md`，忽略裡面『已棄用』那節，其他部分抓重點」。

### 跟相鄰機制的邊界：subagent vs skill

skill 是「方法」：一段指示，預設直接在你目前的對話裡執行，像貼在牆上的操作手冊；subagent 是「執行者」：有自己 context、工具、權限的獨立 agent，像被叫去隔壁房間做事的另一位廚師——手冊是照著做的步驟，廚師才是真正動手的人。

兩者常見的搭配是「skill 裡指示叫 subagent 執行」：skill 加上 `context: fork` 這個 frontmatter 欄位，就是把整份 skill 內容交給你指定的 agent 去跑，過程不佔用主對話 context：

```yaml
---
name: release-checklist
context: fork
agent: doc-summarizer
---
```

反過來，subagent 也能用 `skills` 欄位在啟動時就預先載入某份 skill 的完整內容，讓它一開始就具備該領域知識，不用等執行中才自己現找。兩個方向用的是同一套底層機制，差別只在「誰把內容注入誰」。細節見 [skills](../skills/)。

## 限制與注意

- subagent 從全新、空白的 context 開始，看不到主對話歷史、你已經讀過的檔案。Claude 會寫一段委派訊息交代任務，但如果你有特殊限制（例如「忽略 `vendor/` 目錄」），這類細節不會自動傳過去，需要在委派時重新講一次。
- 平行跑多個 subagent，完成後結果會一起回到主對話——一次開太多、每個都回傳詳細內容，反而會把主對話塞爆，跟原本想節省 context 的目的相反。
- subagent 只在單一 session 裡運作。如果你要的是「多個獨立 session 同時跑、集中在一個畫面監控」，那是 background agents 的範疇，見〈[背景任務與排程](../background-tasks-scheduling/)〉，不是這裡講的 subagent。
- 這是持續演進中的功能（例如背景執行的預設行為在近期版本有調整過），細節可能隨版本變動，實際使用前建議跑 `claude --version` 確認版本、對照官方文件。
