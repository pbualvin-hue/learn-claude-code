`CLAUDE.md` 就像貼在冰箱上的家規紙條——不是每次都要對新來的人再唸一遍，而是寫下來，讓每個進門的人一看就知道這個家怎麼運作。Claude Code 每次開新對話都是「重開機」，不會記得你上次交代過什麼；`CLAUDE.md` 就是解決這件事的機制：你把慣例、偏好、規則寫成一份 Markdown 檔案，Claude 開始工作前會先讀一遍。

除了「你手寫」的 `CLAUDE.md`，Claude Code 還有「自動記憶」（auto memory）：Claude 會自己觀察你的修正與偏好，寫成筆記存在專案的記憶資料夾裡，下次自動載入。可以把 `CLAUDE.md` 想成你訂的規定，自動記憶則是 Claude 自己做的工作筆記。

## 進一步了解

### 放在哪裡，決定誰看得到

| 層級 | 位置 | 用途 |
| --- | --- | --- |
| 使用者層 | `~/.claude/CLAUDE.md` | 你個人的偏好，套用到你電腦上所有專案 |
| 專案層 | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 團隊共用規則，進版控、跟大家一起看 |
| 本機層 | `./CLAUDE.local.md` | 只在這台機器、這個專案生效的個人偏好，**要加進 `.gitignore`**，不進版控 |

這些檔案在對話開始時會被合併載入，載入順序是「範圍大的先讀、範圍小的後讀」，離你目前工作目錄越近的規則，會出現在越後面（也就是離 Claude 決策時越近）。組織也可以部署一份全機適用的「管理層」`CLAUDE.md`，一般個人專案用不到，這裡先跳過。

想快速生成起點，可以在專案目錄跑 `/init`，Claude 會掃描程式庫，自動整理出建置指令、測試方式、專案慣例，寫成 `CLAUDE.md` 草稿讓你調整。

### 三行起步，而不是抄一份大模板

官方建議把 `CLAUDE.md` 控制在 200 行以內——檔案越長，佔用的 context 越多，Claude 照做的準確度反而下降。實務上更好的心法是「三行起步」：不要一開始就把網路上看到的完整範本整份貼進去，而是先只寫 2、3 條你真的會重複用到的規則，之後每次遇到「Claude 又犯了同一個錯」「code review 抓到同樣的問題」，才加一條進去。

寫規則時越具體越好驗證：

```markdown
# CLAUDE.md
- 套件管理用 pnpm，不要用 npm
- 送出前先跑 `pnpm test`，測試沒過不要 commit
- API handler 都放在 `src/api/handlers/`
```

「用 2 格縮排」比「把格式弄好」更容易讓 Claude 照做；「送出前跑 `pnpm test`」比「記得測試」更明確。

### 想在多份檔案間共用內容

`CLAUDE.md` 支援 `@path/to/file` 語法匯入其他檔案，例如 `@README` 會把 README 的內容也載進 context（若只是想在文字裡提到檔名而不匯入，用反引號包住即可，例如 `` `@README` ``）。如果專案已經有 `AGENTS.md` 給其他工具用，可以在 `CLAUDE.md` 裡用 `@AGENTS.md` 匯入，避免兩邊維護重複內容。

## 🛠️ 實戰進階

### 範例一：跨檔案共用內容，同時控制匯入深度

團隊常見情境：專案已經有給其他 AI 工具看的 `AGENTS.md`，不想兩邊維護重複規則，又想在 `CLAUDE.md` 補一些 Claude 專屬指示：

```markdown title="CLAUDE.md"
@AGENTS.md

## Claude 專屬指示
- `src/billing/` 底下的變更一律先進 plan mode 再動手
```

Claude 會先載入 `AGENTS.md` 的內容，再接著讀後面 `## Claude 專屬指示` 這段。若不想額外開一個檔案，也可以用符號連結（`ln -s AGENTS.md CLAUDE.md`）讓兩個檔名指向同一份內容；不過 Windows 上建立符號連結需要系統管理員權限或開發者模式，實務上大多數情況用 `@import` 更省事。

`@import` 支援巢狀：被匯入的檔案裡如果又寫了 `@其他檔案`，Claude 會繼續展開，最深展開 **4 層**；相對路徑是相對「引用它的那個檔案」解析，不是相對你目前的工作目錄，跨資料夾 `@import` 時要注意這點，容易找錯檔。

另一個實務技巧：如果你同時開多個 git worktree，`CLAUDE.local.md` 只存在於建立它的那個 worktree，不會自動出現在別的 worktree。要讓個人偏好跨 worktree 共用，改成從家目錄匯入：

```markdown title="CLAUDE.md"
# Individual Preferences
- @~/.claude/my-project-instructions.md
```

第一次在專案裡遇到「外部匯入」（匯入路徑跑到專案目錄之外，例如上面這個 `~/.claude/...`）時，Claude Code 會跳出一次核准對話框列出要匯入的檔案；按下拒絕之後，這個匯入就會保持關閉，不會再跳出來問——想重新啟用要自己去調整設定，不是下次對話會再問一次。

### 範例二：用 `/context` 和 `/memory` 確認規則真的被載入

規則沒被遵守時，第一步不是懷疑 Claude「不聽話」，而是先確認規則有沒有真的進到 context 裡：

```
/context
```

`/context` 會列出這次對話實際載入了哪些 `CLAUDE.md`／`CLAUDE.local.md`，如果你以為存在的某層檔案沒出現在清單裡，代表 Claude 根本沒看到它——常見原因是檔案放錯層級（例如放進子目錄，子目錄的 `CLAUDE.md` 要等 Claude 讀該目錄下的檔案時才會載入，不是啟動時就讀）。

```
/memory
```

`/memory` 用來瀏覽、打開你所有層級的 `CLAUDE.md`／`CLAUDE.local.md`，也可以切換 auto memory 開關、開啟 auto memory 資料夾看 Claude 自己寫了什麼筆記。簡單分工：**`/context` 用來確認「這次對話載入了什麼」，`/memory` 用來瀏覽和編輯「有哪些檔案存在」**。

若要更細追查是哪個規則檔在哪個時機被載入（例如懷疑某條 `.claude/rules/` 路徑規則沒生效），可以掛 `InstructionsLoaded` hook 把每次載入的檔案記錄下來，這是官方文件建議的除錯手段（進階用法，日常排查通常 `/context` 就夠用）。

### 跟相鄰機制的邊界

- **CLAUDE.md**：你寫的建議，Claude 會盡量照做，但終究是它自己判斷要不要遵守，沒有強制力；適合「規範、慣例、專案架構說明」這類「照做通常沒問題、少數情況需要彈性」的內容。
- **[hooks](../hooks/)**：保證執行，不管 Claude 當下怎麼判斷都會跑；「這件事無論如何都要發生」（例如每次 commit 前一定要跑 lint、危險指令一定要擋下）才該用 hook，寫進 CLAUDE.md 只能算「拜託」。
- **auto memory**：Claude 自己觀察你的修正寫下的筆記，不是你手寫的規則；適合「Claude 從錯誤中學到的事」（除錯技巧、build 指令細節、你隨口糾正過的偏好），你不用手動維護，但也因此不受你直接控制內容怎麼寫。真的希望某件事被穩定記住、措辭精準，還是手動寫進 CLAUDE.md 比較可靠。

### 常見誤用

- **把「一定要做到」的事寫進 CLAUDE.md，期待 100% 遵守**：CLAUDE.md 是 context，不是強制設定，兩份規則互相矛盾時 Claude 可能隨機選一份照做。真正要擋下動作或保證某件事發生，交給 [hooks](../hooks/) 或權限規則，不是加大字體、加驚嘆號寫進 CLAUDE.md。
- **以為拆成多個 `@import` 檔案可以省 context 額度**：拆檔案方便的是「維護組織」——分主題管理、方便多人協作——但所有被匯入的檔案在啟動時一樣會整份載入 context，並不會因為拆開就變輕。真正想省 context，要靠[路徑限定規則](../claude-directory/)（只在碰到特定檔案類型時才載入）或直接刪減內容，而不是分檔案。

## 限制與注意

- **`CLAUDE.md` 不是強制規則，是「context」**：Claude 會盡量照做，但不保證百分之百遵守，尤其是指示模糊或幾份檔案彼此矛盾的時候。如果你要的是「無論如何都要擋下某個動作」，那是 [hooks](../hooks/) 的工作，不是 `CLAUDE.md`。
- 檔案太長會反效果：內容塞太多，Claude 遵從度反而下降，也更吃 context 額度。
- 自動記憶的索引檔 `MEMORY.md` 每次只會載入前 200 行或 25KB，超過的部分要 Claude 主動去讀其他主題筆記檔才會用到，不是全部自動塞進 context。
- 個人化內容公開分享前務必去識別化：`~/.claude/CLAUDE.md` 這類個人偏好檔常常含有真實路徑或私人習慣，貼到公開場合前要先清過。
- `CLAUDE.local.md` 一定要記得加進 `.gitignore`，否則等於把「只給自己看」的內容洩漏給整個團隊。
