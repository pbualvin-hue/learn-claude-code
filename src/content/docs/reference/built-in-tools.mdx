把 Claude Code 想成一位隨身帶著整套工具腰帶的工人：讀檔案有讀檔案的工具、改檔案有改檔案的工具、上網查資料有查資料的工具。
Claude 自己會判斷該用哪一支，不需要你下指令告訴它「用讀檔案工具」。但有些工具動作比較大（例如刪改你的檔案、跑指令），
所以在真的動手之前，Claude Code 會先跳出視窗問你「可以嗎？」——這就是權限系統，它管的其實是「哪支工具可以自己做、哪支要先問過你」。

## 進一步了解

### 常見工具

| 工具 | 做什麼 | 預設要不要先問你 |
| --- | --- | --- |
| `Read` | 讀取檔案內容（含圖片、PDF、Jupyter notebook） | 工作目錄內不用 |
| `Grep` | 用 ripgrep 語法搜尋檔案內容 | 不用 |
| `Glob` | 用檔名 pattern（如 `**/*.ts`）找檔案 | 不用 |
| `Edit` | 對指定檔案做精確字串取代（不是 regex） | 要 |
| `Write` | 建立新檔案或整份覆蓋既有檔案 | 要 |
| `Bash` | 執行 shell 指令 | 要（但 `ls`、`cat`、`git status` 這類唯讀指令不用問） |
| `WebFetch` | 抓取指定網址內容，用小模型先摘要成 Claude 看得懂的重點 | 要（第一次連到新網域時） |
| `WebSearch` | 呼叫網頁搜尋，回傳標題與連結 | 要 |
| `Agent` | 開一個有獨立 context 的 subagent 處理任務 | 不用（但 subagent 內部呼叫工具仍照一般規則被問） |

實際工具清單比這張表長很多（官方 [tools 參考頁](https://docs.claude.com/en/docs/claude-code/tools-reference)列了三十多個，
還有像 `NotebookEdit`、`LSP`、`Skill`、排程相關工具等），而且會依你用的 provider、平台、設定不同而有增減，
不要假設每個人畫面上看到的工具都一樣。

### 權限系統怎麼分類工具

官方把工具分三種等級：

1. **唯讀工具**（Read、Grep、Glob 等）：在你目前的工作目錄內不用問，因為不會改變任何東西；讀工作目錄以外的路徑才會跳提示。
2. **Bash 指令**：預設要問，但有一組內建的唯讀指令（`ls`、`cat`、`pwd`、`git status` 等）不用問，因為它們本來就不會改變系統狀態。
3. **會改檔案的工具**（Edit、Write 等）：預設要問，一旦你按下「不用再問」，這個決定通常維持到這次對話（session）結束。

`/permissions` 指令可以查看、也可以直接調整這些規則（例如寫死「這個專案不用問 `Bash(npm run test *)`」）。

## 🛠️ 實戰進階

### 範例一：Grep → Read → Edit 的典型流程，以及它跟 Bash 讀檔指令的關係

多數「改程式碼」的任務會走這條路：先用 `Grep` 大範圍搜關鍵字定位候選檔案，再用 `Read` 打開確認上下文，最後用 `Edit` 做精確取代。這條順序不是習慣問題，是被強制的——`Edit` 要求同一次對話裡「先讀過才能改」，沒讀過或讀完後檔案被外部改動過，`Edit` 會直接失敗（見下方限制與注意）。

容易漏掉的一點：這條「先讀過」的規則，連帶會套用到用 Bash 讀檔的等價指令上（例如 `cat`、`head`、`tail`、`sed` 讀取檔案內容），也就是說 `Read` 的允許/擋下規則是「盡力而為」套用到這些 Bash 讀檔指令，不是只管 `Read` 工具本身；反過來，`Read` 的 deny 規則也會連帶擋住同路徑的 `Edit`（含在該路徑新建檔案），因為改檔案的最後一步要把結果讀回來確認。

### 範例二：用參數比對縮小 ask 規則的範圍，而不是整支工具都要問

除了工具名稱＋路徑／指令的規則，`deny` 與 `ask` 規則還能直接比對工具呼叫的參數值，語法是 `Tool(參數名:值)`：

```json title=".claude/settings.json"
{
  "permissions": {
    "ask": [
      "Bash(run_in_background:true)",
      "Agent(isolation:worktree)"
    ]
  }
}
```

- `Bash(run_in_background:true)`：只在 Claude 要求把指令丟到背景執行時才問你一次，前景的一般指令不受影響——背景指令你比較難即時看到它在做什麼，多一道確認合理。
- `Agent(isolation:worktree)`：只在 subagent 要求開一個獨立 git worktree 時才問——這個動作會多開一份工作目錄副本，跟單純委派一個唯讀 subagent 風險不同，值得分開處理。

注意這個 `Tool(參數名:值)` 寫法只能用在 `ask`／`deny`，而且只認工具輸入裡「頂層」的參數；像 Bash 的 `command`、Edit/Read/Write 的 `file_path`、WebFetch 的 `url` 這些工具已經有自己專屬比對邏輯的欄位，不吃這套參數比對語法（寫了會在啟動時被警告忽略），這些欄位還是要用各工具自己的規則寫法（`Bash(指令前綴 *)`、`Edit(路徑)`、`WebFetch(domain:網域)`），完整語法見 [settings 與權限模式](../settings-permissions/)。

### 跟相鄰機制的邊界

- **工具權限規則（allow/deny/ask）**：只能比對「這支工具被呼叫時，某個欄位長什麼樣子」——路徑、指令前綴、網域、或上面提到的頂層參數值。比對得到就決定放行/擋下/詢問，比對不到就是不管。
- **[hooks](../hooks/)**：當你要的判斷邏輯超出「比對一個欄位」的範圍——例如要看 `Edit` 準備寫入的內容是不是含有密鑰字串、或要呼叫外部服務查一下這個網址是不是在黑名單——permission 規則的比對語法就不夠用了，這時才需要寫一支 `PreToolUse` hook 腳本自己判斷。同一件「這個指令永遠不准跑」的事，能用一條 deny 規則講清楚，就不必為它多寫一支 hook。

### 常見誤用

- **以為擋掉 `WebFetch` 就等於斷網**：`WebFetch` 的 deny／`domain` 規則只管 `WebFetch` 這支工具，如果 `Bash` 還開著，Claude 一樣能用 `curl`、`wget` 連到任何網址，兩者是分開的關卡，官方文件也直接點名這一點。真的要限制對外連線，`Bash` 那邊也要一起處理（deny 掉 `curl`／`wget` 或改走 sandbox 的網路限制）。
- **以為 `Agent` 工具本身會先問一次**：表格裡 `Agent` 標示「不用先問」，指的是「委派一個 subagent」這個動作本身不用問；subagent 內部實際呼叫 Bash、Edit 這些工具時，還是照你設定的權限規則一支一支問，不會因為委派層多一層就被放行或被多問一次。

## 限制與注意

- **WebFetch 不是原始網頁**：它會先用一個小模型依你給的 prompt 摘要內容，Claude 拿到的是摘要後的結果，不是完整原文。如果它說「頁面沒提到 X」，有可能只是你的 prompt 沒問到 X，不代表原始頁面真的沒有——想拿到未加工內容要改用 `curl`（透過 Bash）。
- **Glob 預設不理會 `.gitignore`，Grep 會**：這是兩者刻意設計成不一樣的地方，用 Glob 找檔案時可能會挖出被 `.gitignore` 排除的檔案。
- **Edit 要求「先讀過才能改」**：Claude 必須在同一次對話中先用 Read（或特定形式的 Bash 指令如 `cat`）讀過那個檔案，且檔案在讀取後沒有被外部改動，才能對它做 Edit；不滿足條件會直接失敗，不是靜默略過。
- **工具名稱大小寫要對**：寫權限規則（如 `Bash(npm run build)`）時用的是這裡列出的正式工具名稱，拼錯或大小寫錯不會生效，還可能在啟動時被 Claude Code 標記為警告。
- **這份清單不含 MCP 工具**：連接外部服務（如 Notion、GitHub）用的是 MCP 伺服器提供的工具，跟這裡講的內建工具是分開的兩層。
