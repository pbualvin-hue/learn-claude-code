[下載練習資料夾](/learn-claude-code/downloads/practice-folder.zip)，解壓縮後你會看到一個很亂的資料夾。

截圖、重複的文件、隨便命名的檔案——很像大部分人桌面真實的樣子。

（只是這份是我們準備好的練習素材，不是你的真實檔案，放心用。）

整個過程你都在跟 Claude Code **對話**，不用自己動手打任何 shell 指令。

全程只在這個練習資料夾裡操作。做壞了就重新解壓縮一次，零成本。

慢慢來——這是刻意設計的，讓你在放心的地方，先練習跟 AI 一起做事的節奏。

## 🎯 第一站：說目標，不說步驟

很多人跟 Claude 講話時，習慣像交代作業一樣把每一步都講清楚（先做這個、再做那個）。

更好的方式，是告訴它你想要的**結果**，讓它自己想辦法，你再確認它的計畫合不合理。

<Steps>
1. 打開 Claude Code，把工作目錄切到你解壓縮出來的練習資料夾。
2. 跟它說：「幫我看一下這個資料夾，想想看要怎麼整理會比較好找檔案，先跟我說你的想法，先不要動手。」
3. 讀它回覆的分類想法（通常會抓到：有重複檔案、檔名亂、可以依類型分類等）。
</Steps>

**你會看到**：Claude 列出資料夾內容，並提出一個整理方案（例如分成幾個子資料夾、哪些看起來是重複檔案建議合併），但還沒有真的動手搬移任何東西。

<details>
<summary>做壞了怎麼辦？</summary>
如果它已經先動手做了才問你，或你不小心答應了一個還沒想清楚的計畫——沒關係，這是練習資料夾。直接刪掉整個資料夾，重新解壓縮 `practice-folder.zip` 就是全新的一份。
</details>

**這一站你沉澱了哪條規則？** 「說目標不說步驟」可以用在其他情境嗎？想一下你平常最容易忍不住把步驟講太細的場景。

## 🔐 第二站：它跳出「這樣可以嗎？」的視窗

<Steps>
1. 跟 Claude 說：「你的方案我同意，開始執行。」
2. 它準備搬移或重新命名檔案時，會跳出一個提示問你要不要允許。
3. 先選允許一次，看著它做完第一批操作。
</Steps>

**你會看到**：類似「Claude 想要執行某個操作，允許嗎？」的提示。

這是 Claude Code 這個工具本身的安全機制，不是「Claude 想不想做」的問題——即使你什麼都沒設定過，它預設就是每種新工具第一次用都會先問你一次。

<details>
<summary>做壞了怎麼辦？</summary>
選錯要允許的動作、結果不如預期，一樣重新解壓縮 `practice-folder.zip` 就好。這正是練習資料夾存在的理由：放心去試允許或拒絕，看看差異在哪裡，不用怕真的搞砸什麼。
</details>

**這一站你沉澱了哪條規則？** 如果你發現自己一直在允許同一種操作（例如在這個資料夾內搬移、重新命名檔案），可以直接跟 Claude 說：「這個資料夾裡搬移、重新命名檔案的操作，以後不用每次都問我。」它會告訴你怎麼記住這個偏好——這就是下一站要學的東西。深入了解：[settings 與權限模式](../../../reference/settings-permissions/)。

## 🧠 第三站：它又要你重講一次

<Steps>
1. 結束這次對話（或想像明天重新打開）。
2. 開一個新的 Claude Code 對話，一樣切到練習資料夾。
3. 問它：「上次我們說好的檔案分類規則是什麼？」
</Steps>

**你會看到**：它答不出來，或者只能靠資料夾現況用猜的——因為每次新對話都是「重開機」，不會記得你上次口頭交代過的事。

卡住很正常，不是它變笨了。解法是把規則寫下來，讓 Claude 每次開始前都會讀到：

<Steps>
1. 跟 Claude 說：「幫我在這個資料夾建一個 `CLAUDE.md`，把我們剛才講好的分類規則寫進去。」
2. 看它產生的內容。
</Steps>

**你會看到**：它建立一個 `CLAUDE.md`，裡面用一兩行寫下具體的規則，例如「截圖都歸類到 `screenshots` 資料夾」——不是一次塞十幾條，先從你真的會用到的那一條開始就好。

<details>
<summary>做壞了怎麼辦？</summary>
`CLAUDE.md` 寫錯或想重寫，直接刪掉這個檔案重新來一次即可，不影響資料夾裡其他檔案。
</details>

**這一站你沉澱了哪條規則？** 寫進 `CLAUDE.md` 的那一條，就是你的答案。深入了解：[CLAUDE.md 與記憶](../../../reference/claude-md-memory/)；想知道怎麼幫自己的專案養一份好用的 `CLAUDE.md`，可以看[打造你自己的 CLAUDE.md](../../your-own-claude-md/)。

## 🔁 第四站：「這串流程我以後還想用」

<Steps>
1. 跟 Claude 說：「如果我以後還有其他雜亂資料夾要整理，有沒有辦法不用每次都重新跟你解釋一次規則？」
2. 聽它解釋這個概念——一份寫好的操作手冊，需要時才叫出來用，不需要時不佔對話空間。
</Steps>

**你會看到**：Claude 說明可以把這套整理流程包成一個 skill，之後用類似 `/整理資料夾` 的指令就能重複叫出來用。

這一站非工程師只要「認識」這個概念就好，不需要自己動手寫。深入了解：[skills](../../../reference/skills/)——之後有興趣自己動手，再回來看細節。

<details>
<summary>做壞了怎麼辦？</summary>
這一站只是認識概念，不需要你真的建立 skill 檔案，所以沒有「做壞」的風險。
</details>

**這一站你沉澱了哪條規則？** 記住這個概念就好——「重複用到的流程，以後可以包裝起來」，不用現在就動手做。

接下來三站，主題從「怎麼整理」轉成「怎麼放心」——學會怎麼後悔、怎麼設定不用每次都問的允許，還有怎麼驗收 Claude 做的事對不對。

## 💾 第五站：先存檔，再放手

<Steps>
1. 跟 Claude 說：「幫我把這個資料夾現在的狀態用 git 存檔一次，之後我做壞了才回得來。」
2. 它可能會先問你要不要允許，看著它把目前的狀態存下來。
3. 接著故意做一件你等一下會後悔的事，跟它說：「把所有檔案全部搬到同一層，不要子資料夾。」
4. 看結果——剛才整理好的分類結構，被打散成一整層。
5. 跟它說：「幫我回到剛才的存檔點。」
</Steps>

**你會看到**：資料夾回到存檔時的樣子——做壞真的可以後悔。

**不用學任何 git 指令，全程用講的**：Claude 自己知道要跑哪些 git 指令，你只需要負責決定「什麼時候該存檔」「要不要回檔」。

<details>
<summary>做壞了怎麼辦？</summary>
如果回檔後的結果不是你預期的樣子（例如只回復了部分檔案），最保險的做法是整個資料夾重新解壓縮 `practice-folder.zip`，重新從第一站存一次檔。這一站本來就是在練習「後悔」，就算這次後悔藥沒吃對，也還有重來一次的空間。
</details>

**這一站你沉澱了哪條規則？** 「動手整理前，先請它 git 存檔一次」——值得變成習慣，甚至寫進 CLAUDE.md。深入了解：[git 是你的後悔藥](../../../trust/git-as-undo/)。

## 🔓 第六站：讓它記住你的允許

接續第二站的情境：如果你發現自己一直在允許同一種操作（搬移、重新命名檔案），每次都要按一次允許，很煩。

<Steps>
1. 跟 Claude 說：「這個資料夾裡搬移、重新命名檔案的操作，以後不用每次都問我。幫我設定好，但範圍只限這個資料夾。」
2. 看它提出要怎麼設定——通常是在專案的設定檔裡加幾條範圍很窄的允許規則。**先看過它要加什麼再答應**，不要看都沒看就說好。
3. 讓它加上去，然後再請它搬移一個檔案試試，看看是不是真的不再跳出確認視窗。
</Steps>

**你會看到**：同類操作不再跳確認視窗，其他還沒設定過的操作（例如刪除檔案）依然會先問你。

:::caution[誠實提醒]
允許規則是把安全關卡打開一個小洞——範圍越窄越好，只放行你真的常用、風險低的操作（搬移、重新命名），「刪除檔案」這類危險操作不要放行。

權限提示視窗本身也有一個「這次不要再問」的選項，但它記多久要看動作種類：跑指令（搬移、重新命名這類操作背後通常是一條指令）選了「不要再問」，會被永久寫進這個資料夾的設定檔，之後開新對話一樣有效；但如果是「編輯檔案內容」，官方文件說這種「不要再問」只會記到這次對話結束為止，不會寫進設定檔。同一顆按鈕，效果不一樣——這也是本站特地請 Claude 把規則直接寫進設定檔的原因：看得到、查得到，比不確定記多久的選項更可靠。
:::

<details>
<summary>做壞了怎麼辦？</summary>
設定加錯、或想拿掉這條允許規則，直接跟 Claude 說：「把剛才那條允許規則刪掉」，或請它打開設定檔手動改。這份設定只影響你這個資料夾，不會動到別的地方。
</details>

**這一站你沉澱了哪條規則？** 這條規則本身就寫進了設定檔（`.claude/settings.local.json`）——順便認識一下：`CLAUDE.md` 是「偏好」，Claude 會盡量照做但不保證；`settings` 裡的權限規則才是「硬規則」，Claude Code 這個工具本身會擋下不符合的操作，不管 Claude 想不想做。深入了解：[settings 與權限模式](../../../reference/settings-permissions/)。

## 🔍 第七站：抓它一次錯

<Steps>
1. 跟 Claude 說：「把資料夾裡所有檔案列成一張清單：檔名、你判斷的類型、放在哪個資料夾。」
2. 自己抽 2～3 個檔案，打開來對照它的清單寫得對不對。
3. 再問它：「這張清單裡，有沒有哪個檔案你其實不太確定？」
</Steps>

**你會看到**：多半能找到小出入——分類見仁見智，或它其實漏了什麼；被追問時，它常會承認某幾個判斷不太確定。

**它回報時的自信程度，跟正確程度是兩回事。** 抽查＋追問，就是最簡單可靠的驗收法；就算這次沒抓到錯也沒關係，驗收的價值在於你「確認過」，不是在於「猜它對」。

<details>
<summary>做壞了怎麼辦？</summary>
這一站只是列清單、看清單，沒有真的動任何檔案，所以沒有「做壞」的風險。清單列錯了，直接請它重新列一次就好。
</details>

**這一站你沉澱了哪條規則？** 「整批處理後，固定抽查 2 個結果再收工」——這條規則幾乎能套用在任何 AI 幫你批次處理的場景。深入了解：[AI 會犯錯——怎麼驗證它的產出](../../../trust/verify-ai-output/)。

## ⚠️ 限制與注意

- AI 整理檔案可能會誤判：它覺得是重複的兩個檔案，實際內容可能不同；它覺得某個檔名沒用想丟去別的分類，對你來說可能很重要。每次操作完，自己花一分鐘看結果對不對，不要整批信任、不檢查。
- 這個軌道全程用練習資料夾，做壞了零成本；但同一套操作換到你真實的檔案時，建議先在小範圍（例如複製一份出來）試過，確認 Claude 的判斷跟你想的一致，再放手讓它處理整批。
- 你是這個專案的負責人，不是旁觀者：Claude 提出的計畫、做完的結果，最終判斷「對不對」的還是你。
