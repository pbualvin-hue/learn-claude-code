[下載範例文件包](/learn-claude-code/downloads/sample-docs.zip)——裡面是我們準備的幾份範例文件，不是你的真實資料。

放心練習「一次丟一堆文件，叫 Claude 幫忙整理摘要」這件事，不用擔心隱私或版權問題。

整個過程你都在跟 Claude Code **對話**，不用自己動手打任何 shell 指令。

:::caution[想改用自己的文件？]
如果你想拿自己的真實文件來練這個軌道，請先讀過[什麼不該給 AI](../../../trust/what-not-to-share/)這一課，確認你要交給 Claude 的文件裡沒有不該外流的內容，再回來繼續。這是本軌道唯一的硬性規定。
:::

## 🎯 第一站：說目標，不說步驟

<Steps>
1. 把工作目錄切到範例文件包所在的資料夾。
2. 跟 Claude 說：「幫我把這些文件的重點整理成一份摘要，我沒空全部看完」——不要一份一份指定要怎麼讀。
3. 看它怎麼回應。
</Steps>

**你會看到**：Claude 先列出資料夾裡有哪些文件，說明打算怎麼進行（例如逐份讀取抓重點、最後彙整成一份摘要檔），再問你要不要開始。

<details>
<summary>做壞了怎麼辦？</summary>
範例文件包是唯讀素材，重新解壓縮 `sample-docs.zip` 就能還原，不影響任何真實資料。
</details>

**這一站你沉澱了哪條規則？** 你只講了想要的結果（「重點摘要」），沒有講「怎麼讀」，但 Claude 自己想出了合理的做法——下次卡在不知道怎麼交代任務時，先試試只講結果。

## 🔐 第二站：它跳出「可以幫你寫這個檔案嗎？」

<Steps>
1. 跟 Claude 說：「開始吧。」
2. 它讀完文件、準備把摘要寫成一個新檔案時，會跳出許可確認。
3. 允許它寫入。
</Steps>

**你會看到**：類似「允許建立 `summary.md` 嗎？」的提示——這是 Claude Code 對「寫入新檔案」這個動作的預設把關，不管你有沒有設定過什麼，它都會先問過一次。

<details>
<summary>做壞了怎麼辦？</summary>
摘要檔案寫壞了，直接刪掉重來，或重新解壓縮 `sample-docs.zip` 拿回乾淨的素材。
</details>

**這一站你沉澱了哪條規則？** 跟 Claude 說：「這個資料夾裡建立摘要檔案的操作，以後不用每次都問我。」它會帶你走一遍怎麼記住這個偏好。深入了解：[settings 與權限模式](../../../reference/settings-permissions/)。

## 🧠 第三站：它又要你重講一次「我要哪種摘要」

<Steps>
1. 開一個新對話，一樣切到範例文件包資料夾。
2. 問它：「上次我們說好摘要要用什麼格式？」
</Steps>

**你會看到**：它答不出來，或用猜的——每次新對話都是「重開機」，不會記得你上次交代的格式偏好（例如「先列行動項目，再列重點，一律條列式」）。

<Steps>
1. 跟 Claude 說：「幫我在這個資料夾建一個 `CLAUDE.md`，把摘要格式的規則寫進去。」
2. 看它產生的內容——通常只會先寫下一兩行你真的會用到的規則。
</Steps>

<details>
<summary>做壞了怎麼辦？</summary>
`CLAUDE.md` 寫錯或想重寫，直接刪掉這個檔案重新來一次即可。
</details>

**這一站你沉澱了哪條規則？** 寫進 `CLAUDE.md` 的那一條摘要格式規則，就是你的答案。深入了解：[CLAUDE.md 與記憶](../../../reference/claude-md-memory/)、[打造你自己的 CLAUDE.md](../../your-own-claude-md/)。

## 🔁 第四站：「這個流程我想每批文件都能用」

<Steps>
1. 跟 Claude 說：「如果我以後常常有一批新文件要摘要，有沒有辦法不用每次都重新交代規則？」
2. 聽它解釋 skill 這個概念——一份寫好的操作手冊，需要時才叫出來用。
</Steps>

**你會看到**：Claude 說明可以把這套摘要流程包成一個 skill，之後用類似 `/摘要文件` 的指令重複叫出來用。這一站非工程師只要「認識」這個概念就好，不需要自己動手寫。深入了解：[skills](../../../reference/skills/)。

<details>
<summary>做壞了怎麼辦？</summary>
這一站只是認識概念，不需要你真的建立 skill 檔案，沒有「做壞」的風險。
</details>

**這一站你沉澱了哪條規則？** 記住這個概念就好——「重複用到的流程，以後可以包裝起來」。

## 🌐 第五站（進階）：文件不在這個資料夾，而是在雲端服務裡呢？

這一站只在你想接外部資料時再看，例如以後這些文件其實放在 Notion 或雲端硬碟裡，不想每次都先下載。

<Steps>
1. 問 Claude：「以後這些文件如果放在 Notion 裡，能不能讓你直接讀，不用我先下載？」
2. 聽它解釋 MCP 連接器的概念。
</Steps>

**你會看到**：Claude 說明可以透過 MCP 連接器接上像 Notion 這樣的外部服務，讓它直接讀取，不用你手動搬資料。非工程師只需要知道：到 Anthropic 已審核過的 Directory 挑一個現成連接器加進來、用 `/mcp` 完成登入即可，**不需要自己動手寫**一個連接器——那是開發者軌道的事。深入了解：[MCP 連接器](../../../reference/mcp/)。

**這一站你沉澱了哪條規則？** 需要接外部資料時，先想「有沒有現成、已審核過的連接器」，不要一開始就想自己寫一個。

## 💾 第六站：先存檔，再放手

<Steps>
1. 產出第一批摘要後，跟 Claude 說：「幫我把現在的成果用 git 存檔一次。」
2. 存檔完成後，故意做一件會後悔的事：跟它說「把所有摘要合併成一個大檔案，原本分開的檔案都刪掉。」
3. 看著合併後的結果，發現你比較想要分開的檔案 → 跟它說：「我不要剛才那個改動了，幫我回到剛才的存檔點。」
</Steps>

**你會看到**：分開的摘要檔案原封不動地回來了，合併、刪除的動作都被撤銷——不用學任何 git 指令，全程用講的。

<details>
<summary>做壞了怎麼辦？</summary>
如果存檔點跑掉、或 Claude 說找不到可以回去的版本，代表這次示範沒抓好時機——重新解壓縮 `sample-docs.zip`，從第一站重新走一次即可，不影響任何真實資料。
</details>

**這一站你沉澱了哪條規則？** 「批次處理前先存檔」——先存過一次檔，後面不管做壞什麼都能回頭。深入了解：[git 是你的後悔藥](../../../trust/git-as-undo/)。

## 🔓 第七站：讓它記住你的允許

情境：每寫一個摘要檔都要允許一次，很煩。

<Steps>
1. 跟 Claude 說：「在這個資料夾裡建立和編輯摘要檔案，以後不用每次都問我。幫我設定，範圍只限這個資料夾。」
2. **先看過它打算加的規則長什麼樣子，覺得合理再答應**——不要看到「可以幫你設定」就反射性同意。
3. 答應之後，再請它補一份摘要，看看這次還跳不跳確認。
</Steps>

**你會看到**：同一類「在這個資料夾寫摘要檔」的操作，之後不再跳出確認視窗。

:::caution[誠實提醒]
- 允許規則等於在安全關卡上開一個小洞，範圍越窄越好——這裡只該放行「這個資料夾裡的摘要檔案」，不是整台電腦的檔案都能寫。
- 刪除類的操作不要放行；這類動作留著每次都問，比較安心。
- 權限提示裡「這個對話不要再問」的效果，不同動作記住的時間長短不一樣：像這裡的「編輯／寫入檔案」，官方文件寫明只記到這次對話結束，換一個新對話又會重新問一次；真的要它以後每次開新對話都不再問，規則得實際寫進設定檔（`.claude/settings.local.json` 之類）才長期有效——這也是這一站請 Claude「幫我設定」、而不是只按過一次確認視窗的原因。
:::

<details>
<summary>做壞了怎麼辦？</summary>
設定的允許範圍不如預期，直接跟 Claude 說：「把剛才加的那條允許規則拿掉」，它會帶你改回設定檔，不影響任何摘要內容。
</details>

**這一站你沉澱了哪條規則？** 常重複做的同一類操作，先看清楚範圍再放行，不要為了圖方便一次把整個工具都打開。深入了解：[settings 與權限模式](../../../reference/settings-permissions/)。

## 🔍 第八站：抓它一次錯

<Steps>
1. 請 Claude 給每一份摘要標上「原始文件檔名」，方便你之後對照。
2. 隨機挑一份摘要，自己打開對應的原始文件讀一遍，對照摘要有沒有漏掉重點、有沒有寫出原文根本沒講的東西。
3. 問它：「這批摘要裡，哪一份你最沒把握？為什麼？」
</Steps>

**你會看到**：摘要通常大致正確，但常常會有取捨或小出入——**摘要是它的理解，不是原文的替身**。被追問「哪一份最沒把握」時，它通常答得出來，會指出某份文件內容比較模糊或跳躍的地方。

重點句：關鍵文件永遠回頭讀一次原文；抽查加追問，是最簡單的驗收法。

<details>
<summary>做壞了怎麼辦？</summary>
這一站不會改動任何檔案，只是讀跟問——沒抓到落差也沒關係，代表這批摘要品質不錯，換一份文件再抽查一次即可。
</details>

**這一站你沉澱了哪條規則？** 「重要文件的摘要，一定回原文抽查一次」。深入了解：[怎麼檢查 AI 的輸出](../../../trust/verify-ai-output/)。

## ⚠️ 限制與注意

- AI 批次摘要可能漏重點、也可能編造原文沒有的內容（幻覺）。務必自己抽查對照原文，尤其是重要的文件——可以參考[驗證 AI 輸出](../../../trust/verify-ai-output/)這一課。
- 批次摘要不是全自動幫你做決定：它能幫你節省第一輪閱讀的時間，但看過摘要是否正確、要不要照著做，責任仍在你身上。
- 範例文件包是通用素材，不含任何真實個資或私人內容；如果你改用自己的文件，記得先看過上面提到的「什麼不該給 AI」再繼續。
