## 🎯 東西備好了，現在開工

上一頁你選好了專案、也做完開工三檢查。這一頁不教新功能，教的是**流程**——同一套順序，套用到你自己的任何專案上都能用。

## 🚀 開工：第一句話怎麼說

跟軌道站 1 一樣的起手式：說目標，不說步驟。回顧[怎麼跟 Claude 說話](../../how-to-talk-to-claude/)裡的技巧，把你想做的事、背景脈絡一次講清楚。

先別急著讓它動手。第一句話請它做這件事：

```text
先幫我看一遍這個資料夾，了解現在的狀況，
提出一個你打算怎麼做的計畫，先不要動手。
```

看過計畫、確認方向沒錯，再放行讓它開始做。這一步能省下不少「做錯方向」的返工。

## 🔁 中途：每一步的小循環

專案做到一半，重複用這個小循環：

<Steps>

1. 它做完一步，回報「做好了」。

2. 花一分鐘驗收——看它給的證據（跑出來的結果、貼出的內容），不是只看它說做完了。細節看[怎麼檢查 AI 的輸出](../../../trust/verify-ai-output/)。

3. 對了就說「繼續」；錯了就具體講「哪裡不對」＋「你要什麼」，不要只說「不對，重做」。

4. 每完成一個階段，請它 git 存檔一次。存檔點是你的後悔藥，看[git 是你的後悔藥](../../../trust/git-as-undo/)。

</Steps>

這個循環會一直重複，直到專案做完。卡在某一步出不去，往下看對照表。

## 🧱 卡住了怎麼辦

| 症狀 | 第一步做什麼 |
|---|---|
| 它一直問我要不要允許 | 這是[權限就是保險絲](../../../trust/permissions-as-insurance/)在起作用，看懂它想做什麼再決定，不要無腦一直按允許 |
| 它又忘記我說過的 | 寫進 [CLAUDE.md](../../your-own-claude-md/)，別每次重講一次 |
| 它很有自信但做錯了 | 按 `Esc` 打斷，具體講哪裡不對、你要什麼 |
| 同一個錯糾正兩次還是錯 | `/clear` 重開一個對話，把教訓寫進第一句提示 |
| 越做越亂想重來 | git 回檔，回到上一個存檔點 |
| 完全不知道它在做什麼 | 叫它先停下來，解釋目前的計畫 |

更多症狀查[疑難排解](../../../troubleshooting/)、想找功能用[情境索引](../../../scenarios/)。

## 🏁 收工：自我驗收清單

<Steps>

1. 把成果對照你開工第一句話寫的目標——做出來的是不是你原本要的那件事。

2. 自己抽查幾個實際檔案，不要只信它給的摘要。

3. 請它列出「這次做了哪些變更」，跟你自己抽查的結果對一下。

4. 最後 git 存檔一次，收工。

</Steps>

## 🌱 收工後：沉澱

這次糾正它兩次以上的事，就是該寫進 CLAUDE.md 的規則（兩次原則）。

下次還想做同類專案，記住的不是這次的細節，是這套「卡關→怎麼問→發現功能→沉澱規則」的流程可以直接重複用。

## ⚠️ 限制與注意

老實講：卡關不會因為走過一次就消失，每個專案卡的點都不一樣。

這套流程給你的是「卡住時知道下一步做什麼」，不是「保證你不會卡關」。

AI 會犯錯，驗收永遠是你的工作，不是它的。

## ✍️ 跟著做

用「中途小循環」推進你的專案，做完第一個階段。

**預期結果**：完成一個看得到的階段成果、至少 git 存檔一次，也可能沉澱出一條新規則。

<details>
<summary>做壞了怎麼辦</summary>

改壞了，先試著跟 Claude 說「這次的改動有問題，幫我回到上一個存檔點」。

如果連存檔點都沒有、或情況比 git 能救的範圍更嚴重，回去解壓開工前的備份，從那裡重新開始。
</details>
