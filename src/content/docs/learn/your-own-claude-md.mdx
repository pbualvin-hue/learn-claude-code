## 🧭 先備知識

- 這是學習路徑最後一站，假設你已經走完[第一個專案](../../learn/first-project/)軌道——特別是「它又要你重講一次」那一站。你在那裡已經寫過第一條規則了，這一課教你把這件事變成習慣。
- [CLAUDE.md 與記憶](../../reference/claude-md-memory/)——如果你還不知道 CLAUDE.md 是什麼、放在哪裡會被讀到，先看那一課，這裡不重講機制。

## 🎯 這一課要幹嘛

跟著做完，你會有一份**只有三條規則**的 CLAUDE.md。

每一條，都對應你真的遇過的問題。

之後怎麼讓它慢慢長大、怎麼修剪，你也會知道。

不用一次到位，慢慢來就好。

## 📝 CLAUDE.md 是什麼（一句話講完）

它就是你寫給 Claude 的備忘紙條。

貼在每次對話開頭，讓它不用每次都重新猜「你喜歡怎樣」。

想知道細節（放哪裡、怎麼被載入、跟自動記憶差在哪）？看 [CLAUDE.md 與記憶](../../reference/claude-md-memory/)，這裡不重講。

## 🚫 別照抄別人的大模板

網路上常見「完整 CLAUDE.md 範本」，動輒兩三百行。

看起來一次到位、複製貼上很省事。

老實說：**這是反模式，不是捷徑。**

- 別人的規則對應的是別人踩過的坑。跟你的專案、你的習慣不一定相關，甚至互相矛盾。
- 檔案越長，Claude 照做的準確度反而越低——官方建議控制在 200 行以內。塞滿你用不到的規則，只會稀釋掉真正重要的那幾條。
- 規則一旦膨脹，通常沒人會回頭修剪。過時或互相矛盾的規則放著不管，Claude 遇到兩條打架的指示時只會隨便選一條，行為反而更難預測。

大模板的問題不是「內容錯」，是「跟你無關」。

你需要的是你自己的三條，不是別人的三百行。

## 🌱 三行起步：怎麼挑

不要坐下來空想「我應該需要哪些規則」。

回頭問自己：**最近一次你覺得「它怎麼又這樣」，是什麼時候？**

那就是你第一條規則的來源。

<Steps>

1. 只寫真的發生過的問題，不要預防性地猜。

2. 具體到可以驗證：「用 2 格縮排」比「把格式弄好」明確；「送出前先給我看過」比「小心一點」明確。

3. 三條就好，不夠再加，不用一次憑空生出十條。

</Steps>

先從一條開始就好，另外兩條想不到也沒關係，之後再補。

## ✂️ 規則怎麼長大：三個心法

CLAUDE.md 不是寫一次就定案的文件，它是會跟著專案一起呼吸的東西。

<CardGrid>
  <Card title="兩次原則" icon="list-format">
    同一個問題第一次發生，當場糾正 Claude 就好，先別急著寫進去。

    第二次發生，才值得寫成一條規則。這樣可以濾掉只是一次性意外的狀況。
  </Card>
  <Card title="刪除測試" icon="error">
    定期回頭看每一條規則，問自己：「刪掉這條，Claude 會不會因此犯錯？」

    答案是不會，就刪掉。不要因為「當初想很久才寫的」捨不得留。
  </Card>
  <Card title="定期修剪" icon="pencil">
    規則會過期：換了工具、換了偏好、專案階段不同了。

    建議每次開新專案、或每季挑個時間，花五分鐘重讀一遍，過期的直接刪。
  </Card>
</CardGrid>

:::caution
CLAUDE.md 的內容會被讀進 Claude 的 context，公開 repo 更會被任何人看到。

密碼、API key、個人資料絕對不要寫進去——這件事優先於任何規則管理心法。

詳見 [不該分享的東西](../../trust/what-not-to-share/)。
:::

## 👥 雙讀者版本：起步三條規則怎麼挑

你關心的問題，跟你是不是每天寫程式有關。

兩邊起步的重點不太一樣，選你那邊看就好：

<Tabs>
  <TabItem label="非工程師">
    你在意的通常是「溝通方式」，不是程式細節。

    想想你最近幾次糾正 Claude 的回應方式，例如：

    ```markdown
    # CLAUDE.md
    - 回答先講結論，再說原因，不要一開始就落落長解釋
    - 遇到我可能看不懂的技術詞，順便用一句話白話翻譯
    - 任何要送出、發布、花錢的動作，做完前先給我確認，不要自己執行
    ```

    這三條分別對應「回應風格」「輸出格式」「行為邊界」——都是溝通類問題，不是程式規範。
  </TabItem>
  <TabItem label="開發者">
    你在意的通常是「程式怎麼寫、怎麼測、放哪裡」。例如：

    ```markdown
    # CLAUDE.md
    - 套件管理一律用 pnpm，不要用 npm 或 yarn
    - 新增功能前先寫測試，不要先寫實作再補測試
    - API handler 統一放在 src/api/handlers/，不要散落在別處
    ```

    這三條分別對應「工具慣例」「開發流程」「專案結構」——都是你在 code review 裡已經講過不只一次的事，不是憑空想像的規範。
  </TabItem>
</Tabs>

兩邊差異只在於「你會被糾正的地方不一樣」。

起步心法完全相同：三條、真實發生過、具體可驗證。

## ✍️ 跟著做

現在就打開你的專案（或個人層 `~/.claude/CLAUDE.md`），寫出屬於你的前三條規則。

用這三個問句引導自己：

<Steps>

1. 最近一次你覺得「它怎麼又這樣」是什麼事？把它寫成一條具體規則。

2. 有沒有哪件事你已經跟 Claude 講過兩次一模一樣的糾正？那一定要收進來。

3. 有沒有哪個「行為邊界」是你希望 Claude 每次都先問過你、而不是自己執行的？

</Steps>

寫完存檔，開一個新對話，故意做一件會觸發你剛寫的規則的事，看看 Claude 這次會不會直接照做，不用你再糾正一次。

**預期結果**：下次遇到同類情境，Claude 的第一次回應就符合你的規則，不需要你再重複糾正——這就是規則生效的可觀察訊號。

<details>
<summary>做壞了怎麼辦：規則寫太多、太死怎麼收</summary>

如果過一陣子發現 CLAUDE.md 越寫越長、Claude 開始「該做的沒做、不該管的管太多」，用刪除測試逐條檢查：這條刪掉會不會讓 Claude 犯錯？不會就刪。

兩條規則互相矛盾時（例如一條說「先問過我」、另一條說「直接做」），保留最近、最具體的那條，刪掉舊的。

不確定哪條該留？直接問 Claude：「這幾條規則有沒有互相矛盾？」讓它幫你抓出來，你再決定去留。

</details>

## 🔑 該進你的 CLAUDE.md 嗎？

這一課本身就是在做「該進你的 CLAUDE.md 嗎」這件事——只是把它從一句結尾提問，變成一次完整練習。

往後每一課的「跟著做」結尾，都會再問你這句話。

把它當成習慣，不是這一課獨有的任務。

今天寫下的三條規則，過一週後回頭看：有沒有哪一條其實不常用到？有沒有新的「它怎麼又這樣」該補上？

慢慢調整就好，不用急著一次寫對。
