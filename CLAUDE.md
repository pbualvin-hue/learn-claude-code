# CLAUDE.md — learn-claude-code（專案層）
# 繁體中文 Claude Code 學習網站：課程式學習路徑＋字典式功能參考，自動草稿、人工審核、持續跟版
# 版本：1.3（2026-07-11）｜1.1 學習功能（第 6 輪）；1.2 個人化起步（第 7 輪）；1.3 專案驅動＋誠實教學＋信任教育（第 8 輪）

## 一、專案概述
做什麼：Claude Code 官方功能的原創繁中教學網站（學習路徑＋功能參考＋版本更新解讀＋實戰情境索引）。
跑在哪：Astro + Starlight 靜態站，GitHub Actions 建置，GitHub Pages 部署（$0）。
給誰用：使用者本人（進階使用者）＋親友（含完全沒寫過程式的人）。

## 二、已定案決策（Non-Negotiables）
# 每條格式：決策＋（理由）＋必要時附「已駁回的替代案」
# 修改任何一條前必須先提出討論
1. 內容一律原創教學（自己的理解、範例、情境）＋連結回官方文件——公開網站整段翻譯搬運官方文件有著作權風險；駁回：全文翻譯官方文件——版權邊界，第一輪已攤牌。
2. 優先序：內容正確 ＞ 維護成本 ＞ 花費（使用者 2026-07-11 聲明）——所有取捨用這把尺量。
3. 自動草稿必經人工審核（PR review + merge）才發布——錯的教學比沒有教學更糟；駁回：全自動發布——錯誤內容會直接公開給親友。
4. 每頁 frontmatter 必記 `base_version`（該頁內容依據的 Claude Code 版本）與 `sources`（官方文件連結）；官方文件查不到、僅靠推測的行為描述一律標「⚠ 未驗證」——內容正確是第一優先序，可追溯性是它的執行機制。
5. 個人設定／習慣類內容進站前過去識別化檢查清單：無含使用者名稱的路徑、無 email、無私人專案名稱、無硬體環境細節——全站公開。
6. 所有內容為 Markdown/MDX 純文字，不寫進框架元件裡——內容與框架解耦，框架可換、內容可攜（退場機制）。
7. 技術棧鎖定：Astro + Starlight（內建 Pagefind 搜尋、深色模式、側欄結構）＋ GitHub Pages——維護成本與花費最低的組合；駁回：Docusaurus（React 生態較重，維護成本高）、自架 server 或資料庫（靜態內容用不到，違反花費與維護成本優先序）。
8. Repo 設為 **public**——GitHub Free 方案的 Pages 僅支援 public repo（查證 2026-07-11）；本條明確覆蓋全域層「新建 repo 預設 private」規則，內容本來就是要公開的。
9. 更新管線的核心是一個可手動觸發的專案 skill（/update-content），排程只是它的觸發器之一——平台排程功能若變動，手動路徑照常可用（降級路徑內建）。
10. 誠實教學：每個功能頁必含「限制與注意」節，教材不誇大優點、缺陷優先講清（讀者最在意的正是缺陷）；網站對讀者的承諾邊界誠實標示——保證「完成引導軌道專案」，不承諾「你的理想專案會完成」；資安與風險類內容列入審核必抽清單（寫錯的安全感比不寫更糟）。
11. 課程主軸是專案驅動（做中學）：功能在範本專案遇到問題時才登場（just-in-time），不做「先背完所有功能再開始」的課綱式教學；非工程師軌道止於 CLAUDE.md＋安全使用現成連接器，自建 skill／MCP／外掛屬開發者軌道。

## 三、架構分層
```
內容層   content/*.mdx（原創繁中教學，frontmatter 帶 base_version）
框架層   Astro + Starlight（側欄、搜尋、深淺色）
建置層   GitHub Actions（push to main → build → deploy）
部署層   GitHub Pages（public repo，免費）
更新管線  排程 agent（週）→ 讀 changelog → 草稿 draft PR → 人工審核 → merge → 自動部署
```

## 四、關鍵技術規則
- 兩層深度：每頁先寫「入門」段（非工程師可讀，用比喻），進階細節放後段或摺疊區——最弱使用者是完全沒碰過終端機的人。
- 指令與功能名稱保留英文原文（`/config`、hooks、MCP），內文解釋用繁中——搜尋主要靠英文指令名，繞開中文斷詞弱點。
- 草稿審核程序：每次 merge 前隨機抽 3 條功能描述與官方文件比對（黃金抽查，內容正確的可量測機制）。
- 版本落後可視化：網站 footer 顯示「內容基準版本 vX.Y.Z｜最後更新日期」——排程靜默失敗時一眼看得出內容過時。

### 學習體驗規則（UX，2026-07-11 第 6 輪迭代修入）
- 每課結尾必附「跟著做」任務：可直接複製的操作＋預期會看到的結果＋「做壞了怎麼辦」摺疊區；練習以 Claude Code 對話內操作為主（避開 shell 平台差異），必須用 shell 時 Windows／macOS 雙標注。
- 疑難排解頁標題用症狀人話（例：「它一直問我要不要允許」），技術術語放內文——非工程師是用症狀搜尋的，不是用術語。
- 架構圖與流程圖一律文字源碼繪製（Mermaid／SVG）：可版控 diff、深淺色自適應、手機可縮放；禁止點陣圖當示意圖（真實介面截圖除外，截圖需標注擷取時的版本）。
- 無死路原則：每頁頂部「先備知識」（連去該先讀的頁）、底部「下一步」導引；搜尋無結果時導向情境索引。
- 首頁雙入口分流：「我是新手→學習路徑」「我要查東西→搜尋＋情境索引」，兩種讀者不混流。
- 互動狀態（進度勾選、quiz 成績）只用 localStorage，UI 明示「進度僅存於這台裝置」；不做帳號系統（維護成本與隱私）。
- 決策比較頁列入更新管線的強制複審清單：changelog 涉及 hooks／skills／agents／settings／權限任一時，草稿 PR 必須包含決策頁複審結果。
- 個人化起步原則（2026-07-11 第 7 輪修入）：教 CLAUDE.md／規則類內容一律「三行起步」——規則必須對應學習者真實遇過的問題；禁止提供大而全模板讓人整份照抄（照抄他人規則與規則膨脹是反模式，教材要明講）；產生器類工具輸出上限 10 條。個人化是貫穿課程的習慣：每課「跟著做」結尾固定問「這個功能有沒有該進你 CLAUDE.md 的東西？」。

## 五、已知限制（接受，不視為 bug）
- Pagefind 中文斷詞有限：中文長句搜尋效果普通；以英文指令名搜尋＋情境索引頁補償。
- GitHub Pages 100GB/月流量與排程 agent 訂閱額度均為「非合約保證」的免費層——被收回時的備援見 PLAN.md 風險節。
- 內容更新頻率上限為每週一次排程＋手動觸發，不做即時跟版。

## 六、與全域層的關係
- 覆蓋全域「新建 repo 預設 private」→ 本專案 public（Non-Negotiable #8，使用者已裁決）。
- rules/testing.md 對內容（.mdx）不適用；對管線腳本與 Astro 設定適用「build 通過＋Phase 0 驗證項」等效標準。
- agent 委派照全域十二區：內容草稿屬長內容處理，委派 agent 隔離 context，不在主對話直接跑。
